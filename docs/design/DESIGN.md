
# DESIGN.md

AI 에이전트가 웹사이트 프론트엔드를 만들 때 참고하는 사용자 중심 디자인 지침서다.
이 문서는 특정 브랜드의 외형을 복제하기 위한 문서가 아니다.
이 문서는 웹사이트가 사용자의 목표를 더 쉽고, 명확하고, 신뢰감 있게 달성하도록 만드는 원칙을 정의한다.

---

## 0. AI가 항상 확인할 체크리스트

- 사용자가 이 페이지에서 무엇을 이해하고 끝내야 하는지 먼저 정의한다.
- 한 페이지에는 하나의 대표 목적과 하나의 주 CTA를 둔다.
- 복잡한 정책, 절차, 조건은 사용자가 이해할 수 있는 구조와 문장으로 정리한다.
- 중요한 조건, 비용, 제한, 결과, 데이터 사용 방식은 숨기지 않는다.
- 거절, 취소, 뒤로 가기, 나중에 하기, 삭제, 해지 경로를 공정하게 제공한다.
- 색상, 카드, 아이콘, 애니메이션보다 정보 구조, 문구, 접근성, 상태 설계를 우선한다.
- 데스크톱, 노트북, 태블릿, 모바일 웹에서 같은 목적과 흐름이 유지되게 한다.
- 로딩, 빈 상태, 오류, 성공, 비활성 상태를 반드시 설계한다.

---

## 1. 문서 목적

이 문서는 AI가 범용 웹사이트 UI를 만들 때 다음을 구현하도록 돕는다.

- 사용자가 첫 화면에서 “무엇을 할 수 있는지” 바로 이해하는 웹사이트
- 복잡한 내용을 사용자가 덜 고민하도록 정리한 웹사이트
- 전환을 강요하지 않고, 사용자의 선택권을 존중하는 웹사이트
- 쉬운 말, 짧은 문장, 명확한 버튼 문구를 사용하는 웹사이트
- 웹의 넓은 화면, 스크롤, 키보드, 마우스, URL, 검색, 공유 맥락을 고려한 웹사이트
- 모바일 앱 패턴을 그대로 복제하지 않고 웹에 맞게 재해석한 웹사이트
- 접근성, 반응형, 상태 피드백, 신뢰 정보를 기본값으로 갖춘 웹사이트

이 문서는 다음을 요구하지 않는다.

- 특정 브랜드 색상 사용
- 특정 로고, 폰트, 그래픽 자산 사용
- 특정 산업 전용 UI 복제
- 모바일 앱 화면을 웹에 그대로 이식
- 예쁜 카드와 장식 중심의 화면 구성

이 문서의 목표는 다음이다.

> 사용자가 복잡한 일을 단순하게 느끼고, 불안 없이 이해하고, 실수 없이 행동을 끝낼 수 있는 웹사이트를 만든다.

## 2. 핵심 정의

### 2.1 Simplicity

Simplicity는 요소가 적은 화면이 아니다.
Simplicity는 서비스나 시스템이 가진 복잡함을 설계자가 정리해, 사용자에게는 필요한 판단과 행동만 명확하게 전달하는 것이다.

Simplicity는 다음을 의미한다.

- 사용자가 배워야 할 것을 줄인다.
- 사용자가 기억해야 할 것을 줄인다.
- 사용자가 비교해야 할 선택지를 정리한다.
- 내부 정책, 기술 구조, 예외 조건을 사용자 언어로 바꾼다.
- 사용자가 지금 상황과 다음 행동을 예측할 수 있게 한다.
- 중요한 정보는 숨기지 않고, 덜 중요한 정보는 적절한 시점에 보여준다.
- 사용자가 실수해도 회복할 수 있게 한다.
- 정보가 적은 것이 아니라 올바른 순서로 제시되는 상태를 만든다.

Simplicity는 정보 은폐가 아니다.
Simplicity는 정보 정리다.

### 2.2 사용자 중심 UI

사용자 중심 UI는 조직 내부 구조, 기술 구현, 마케팅 목표보다 사용자의 목표와 불안을 먼저 고려한다.

사용자는 보통 다음을 원한다.

- 빠르게 이해하고 싶다.
- 내가 무엇을 해야 하는지 알고 싶다.
- 실수하고 싶지 않다.
- 불필요한 입력을 하고 싶지 않다.
- 내 선택을 되돌릴 수 있는지 알고 싶다.
- 중요한 조건이 숨겨지지 않기를 원한다.
- 거절하거나 나갈 수 있는 선택권을 원한다.

### 2.3 인지 부하

인지 부하는 사용자가 화면을 이해하기 위해 쓰는 정신적 노력이다.

좋은 UI는 다음을 줄인다.

- 읽어야 하는 양
- 기억해야 하는 정보
- 비교해야 하는 선택지
- 추측해야 하는 다음 행동
- 불안하게 만드는 불확실성
- 복구 방법을 모르는 실수

### 2.4 Honest Design

Honest Design은 사용자를 설득하기 전에 이해시킨다.

- 선택의 결과를 숨기지 않는다.
- 거절, 취소, 삭제, 해지를 어렵게 만들지 않는다.
- 중요한 조건을 작은 글씨, 툴팁, 약관 뒤에만 숨기지 않는다.
- 사용자의 불안, 죄책감, 조급함을 이용하지 않는다.
- 사용자에게 불리할 수 있는 정보도 결정 전에 보여준다.
- 사용자가 원하지 않는 설정을 기본값으로 몰래 켜지 않는다.

### 2.5 Web-first

Web-first는 모바일 화면을 넓히는 것이 아니다.
웹의 맥락을 활용해 정보 구조를 더 명확하게 보여주는 것이다.

웹사이트는 다음을 고려해야 한다.

- 넓은 화면
- 긴 스크롤
- 상단 내비게이션
- 사이드바
- 표와 비교 UI
- 키보드와 마우스
- 브라우저 뒤로 가기
- URL 공유
- 새 탭 열기
- 검색과 필터
- 문서 읽기
- 여러 탭 비교

## 3. AI 의사결정 우선순위

AI는 UI를 만들 때 아래 순서로 판단한다.

1. 사용자 목적
2. 페이지의 대표 목적
3. 핵심 정보
4. 핵심 행동
5. 사용자 불안 해소
6. 정보 구조
7. UX writing
8. 다크패턴 방지
9. 접근성
10. 반응형 동작
11. 시각적 분위기
12. 세부 스타일

절대 다음 순서로 만들지 않는다.

1. 예쁜 카드부터 만든다.
2. 임의의 색을 고른다.
3. 아이콘을 많이 배치한다.
4. 애니메이션을 넣는다.
5. 나중에 문구를 채운다.
6. 마지막에 접근성을 추가한다.

좋은 생성 순서:

1. 페이지 목적을 한 문장으로 정한다.
2. 사용자가 가장 먼저 알아야 할 정보를 정한다.
3. 사용자가 가장 먼저 할 행동을 정한다.
4. 꼭 필요한 정보와 나중에 보여도 되는 정보를 나눈다.
5. 제목, 설명, CTA 문구를 먼저 쓴다.
6. 오류, 빈 상태, 로딩, 성공 상태를 설계한다.
7. 레이아웃과 컴포넌트를 선택한다.
8. 접근성과 반응형 규칙을 적용한다.
9. 마지막으로 색상, 그림자, 모션, 장식을 정리한다.

## 4. 웹사이트 전체 구조 원칙

### 4.1 기본 페이지 구조

대부분의 웹사이트는 다음 구조를 가진다.

```text
Header
  - 서비스명 또는 로고
  - 주요 내비게이션
  - 보조 액션
  - 대표 CTA

Main
  - 페이지 제목 또는 Hero
  - 핵심 설명
  - 주요 콘텐츠
  - 보조 콘텐츠
  - 신뢰를 돕는 정보
  - FAQ 또는 추가 안내
  - 반복 CTA

Footer
  - 서비스 정보
  - 정책 링크
  - 문의 경로
  - 보조 내비게이션
````

### 4.2 집중형 흐름 구조

회원가입, 신청, 예약, 문의, 설정, 업로드처럼 사용자가 특정 과업을 끝내야 하는 페이지는 더 단순해야 한다.

```text
Minimal Header
  - 서비스명
  - 뒤로 가기 또는 닫기
  - 진행 상태

Main
  - 현재 단계 제목
  - 짧은 설명
  - 입력 또는 선택
  - 도움말
  - 오류 메시지
  - 다음 행동

Optional Aside
  - 넓은 화면에서 요약, 도움말, 미리보기 제공
```

### 4.3 Header

헤더는 길 찾기를 돕는 장치다.
브랜드를 과시하는 장식 영역이 아니다.

원칙:

* 메뉴 항목 수를 줄인다.
* 메뉴명은 사용자의 목적 중심으로 쓴다.
* 대표 CTA는 하나만 강하게 둔다.
* 로그인, 도움말, 문의 같은 보조 액션은 약하게 둔다.
* 고정 헤더는 사용할 수 있지만 본문을 과도하게 가리지 않는다.
* 좁은 화면에서는 메뉴를 접어도 대표 CTA와 현재 위치는 유지한다.
* 집중형 흐름에서는 전체 메뉴를 줄여 사용자의 집중을 보호한다.

좋은 예:

```text
[서비스명] [기능] [활용 사례] [가이드]      [로그인] [시작하기]
```

나쁜 예:

```text
[로고] [회사소개] [비전] [뉴스] [이벤트] [블로그] [자료실] [문의] [데모] [가입]
```

### 4.4 Footer

푸터는 신뢰와 보조 탐색을 위한 영역이다.

포함할 수 있는 정보:

* 운영 주체 정보
* 문의 경로
* 도움말 링크
* 이용 조건
* 개인정보 관련 안내
* 접근성 안내
* 상태 또는 공지 링크
* 보조 내비게이션

푸터는 링크를 많이 넣는 곳이 아니다.
사용자가 필요할 때 찾을 수 있어야 하는 정보를 정리하는 곳이다.

## 5. 정보 구조 원칙

### 5.1 한 페이지에는 하나의 대표 목적이 있어야 한다

웹사이트는 앱보다 정보량이 많을 수 있다.
하지만 한 페이지의 대표 목적은 하나여야 한다.

나쁜 예:

```text
한 페이지에서 소개, 이벤트, 가입, 자료 요청, 설문, 공지를 모두 같은 강도로 요구한다.
```

좋은 예:

```text
이 페이지의 대표 목적은 서비스 가치를 이해하고 시작 여부를 판단하게 하는 것이다.
보조 정보는 그 판단을 돕는 위치에 둔다.
```

### 5.2 결론을 먼저 보여준다

나쁜 구조:

```text
배경 설명
운영 방식
상세 조건
예외 사항
마지막에 행동
```

좋은 구조:

```text
무엇을 할 수 있는지
왜 유용한지
어떻게 시작하는지
알아야 할 조건
더 자세한 설명
```

### 5.3 정보는 중요도 순서로 배치한다

기본 순서:

1. 현재 페이지가 무엇인지
2. 사용자가 얻는 가치
3. 사용자가 할 수 있는 행동
4. 판단에 필요한 핵심 근거
5. 선택 전 알아야 할 조건
6. 세부 설명
7. 예외 사항
8. 추가 도움말

### 5.4 한 섹션에는 하나의 메시지만 둔다

나쁜 섹션:

```text
서비스 소개 + 이벤트 안내 + 문의 유도 + 기능 비교 + 공지
```

좋은 섹션:

```text
이 섹션은 사용자가 이 기능이 어떤 문제를 해결하는지 이해하게 한다.
```

### 5.5 요약과 상세를 나눈다

* 핵심 요약은 먼저 보여준다.
* 상세는 접힘 영역, FAQ, 문서, 보조 섹션으로 제공한다.
* 사용자의 결정에 영향을 주는 조건은 너무 깊이 숨기지 않는다.
* 복잡한 설명은 표, 단계, 예시로 풀어준다.
* 단어보다 구조로 이해시키는 것을 우선한다.

### 5.6 길 찾기

사용자는 항상 다음을 알아야 한다.

* 내가 어디에 있는지
* 무엇을 할 수 있는지
* 어디로 돌아갈 수 있는지
* 필터나 검색이 적용되었는지
* 다음 단계가 무엇인지
* 문제가 생기면 어디에서 도움을 받을 수 있는지

깊은 구조에서는 breadcrumb, 사이드 내비게이션, 현재 메뉴 표시, 검색 결과 요약을 사용한다.

## 6. 레이아웃 원칙

### 6.1 시선 흐름

기본 시선 흐름:

```text
상단: 현재 위치와 페이지 목적
중앙: 핵심 메시지
그 아래: 근거, 입력, 선택지
행동 위치: CTA
하단: 보조 설명, 도움말, 신뢰 정보
```

AI는 요소를 예쁘게 흩뿌리지 않는다.
사용자의 시선이 자연스럽게 이동하도록 배치한다.

### 6.2 콘텐츠 폭

* 긴 본문은 적절한 읽기 폭으로 제한한다.
* 폼은 너무 넓게 만들지 않는다.
* 옵션 비교나 데이터 화면은 넓은 폭을 사용할 수 있다.
* 대시보드는 넓은 화면을 활용하되 핵심 정보의 우선순위를 유지한다.
* Hero는 넓게 쓸 수 있지만 문장은 짧게 유지한다.

### 6.3 여백

여백은 장식이 아니라 이해를 돕는 도구다.

역할:

* 관련 있는 요소를 묶는다.
* 관련 없는 섹션을 분리한다.
* 제목과 설명의 관계를 드러낸다.
* CTA를 잘 보이게 한다.
* 사용자가 어디부터 읽어야 하는지 알려준다.

규칙:

* 관련 있는 요소는 가까이 둔다.
* 다른 주제는 충분히 떨어뜨린다.
* 섹션 간 여백은 요소 간 여백보다 크게 둔다.
* 카드 내부 여백은 일관되게 둔다.
* 버튼 주변에는 실수 없이 누를 수 있는 공간을 둔다.
* 여백을 늘려 정보를 숨기거나 페이지를 불필요하게 길게 만들지 않는다.

### 6.4 정렬

* 기본은 좌측 정렬을 사용한다.
* 긴 본문은 중앙 정렬하지 않는다.
* 중앙 정렬은 Hero, 빈 상태, 완료 상태처럼 짧은 메시지에만 제한적으로 쓴다.
* 숫자나 상태는 비교하기 쉽게 정렬한다.
* 여러 카드가 있을 때 제목, 설명, CTA 위치를 맞춘다.
* 폼 라벨과 입력 필드의 정렬을 일관되게 유지한다.
* 장식적 비대칭 배치는 가능하지만 정보 이해를 방해하면 안 된다.

### 6.5 화면 밀도

페이지 유형에 따라 밀도는 달라진다.

| 페이지 유형    | 권장 밀도                 |
| --------- | --------------------- |
| 랜딩 페이지    | 낮은 밀도, 메시지 중심         |
| 소개 페이지    | 섹션별 명확한 호흡            |
| 신청·문의 흐름  | 낮은 밀도, 입력과 설명 중심      |
| 옵션 비교 페이지 | 중간 밀도, 비교 명확성 중심      |
| 대시보드      | 중간~높은 밀도, 핵심 상태 우선    |
| 관리자 화면    | 높은 밀도 가능, 단 위계와 필터 명확 |
| 문서·아티클    | 읽기 편한 폭과 줄간격          |
| 도움말 페이지   | 검색과 분류 중심             |

## 7. 반응형 웹 원칙

반응형은 모바일에 맞추는 것이 아니다.
반응형은 다양한 화면에서 같은 목적과 이해 흐름을 유지하는 것이다.

### 7.1 넓은 화면

넓은 화면에서는 다음을 활용할 수 있다.

* 2열 레이아웃
* 우측 요약 패널
* 사이드 내비게이션
* 고정 목차
* 비교 표
* 보조 설명 영역
* 미리보기 패널
* 필터 패널
* 데이터 카드 그리드

좋은 구조:

```text
왼쪽: 주요 내용 또는 입력
오른쪽: 요약, 도움말, 미리보기
문맥 근처 또는 하단: 대표 CTA
```

나쁜 구조:

```text
왼쪽: 메뉴
중앙: 입력
오른쪽: 추천
상단: 배너
하단: 다른 CTA
팝업: 별도 제안
```

### 7.2 노트북

* 세로 공간을 낭비하지 않는다.
* 고정 헤더가 너무 높지 않게 한다.
* 대시보드는 핵심 카드 수를 줄여 첫 화면 집중도를 높인다.
* 표는 주요 열을 우선하고 보조 열은 상세로 보낼 수 있다.
* 모달 높이는 화면을 넘지 않게 한다.
* 필터 패널은 접을 수 있게 한다.

### 7.3 태블릿

* 2열이 답답하면 1열로 전환한다.
* 사이드바는 드로어, 상단 탭, 드롭다운으로 바꿀 수 있다.
* 버튼과 입력 요소는 터치 가능한 크기를 유지한다.
* hover에만 의존하지 않는다.
* 표는 핵심 열 중심으로 재구성한다.

### 7.4 모바일 웹

모바일 웹은 데스크톱 기능의 축소판이 아니다.

원칙:

* 한 화면에 하나의 중심 과제를 더 강하게 유지한다.
* 헤더 항목을 줄인다.
* 중요한 CTA는 숨기지 않는다.
* 긴 표는 카드형 요약이나 주요 열 우선 표시로 바꾼다.
* 사이드바는 접힘 영역, 드로어, 별도 메뉴로 전환한다.
* 보조 설명은 접거나 본문 아래로 이동한다.
* 고정 요소가 본문을 가리지 않게 한다.
* 모달보다 전체 페이지 전환이나 인라인 확장을 우선 고려한다.
* 브라우저 뒤로 가기와 상태 보존을 고려한다.

### 7.5 화면 크기가 바뀌어도 유지해야 할 것

* 페이지의 대표 목적
* 핵심 정보
* 핵심 CTA
* 정보 위계
* 사용자 선택권
* 오류 해결 방법
* 접근 가능한 이름과 상태
* 키보드 이동 순서
* 시각적 안정감

## 8. 비주얼 원칙

### 8.1 분위기

좋은 웹사이트의 분위기는 다음에 가깝다.

* 차분함
* 명확함
* 빠른 이해
* 높은 신뢰감
* 과하지 않은 친근함
* 깔끔하지만 차갑지 않음
* 기능적이지만 딱딱하지 않음
* 장식보다 목적 중심
* 광고처럼 보이지 않음
* 사용자를 압박하지 않음

### 8.2 시각적 절제

줄여야 할 것:

* 불필요한 선
* 불필요한 박스
* 불필요한 그림자
* 불필요한 배지
* 불필요한 아이콘
* 불필요한 구분선
* 불필요한 강조색
* 불필요한 장식 이미지
* 불필요한 움직임
* 불필요한 카드 중첩

다만 구분이 필요한 곳에서는 여백, 배경 면, 타이포그래피, 약한 경계선을 사용한다.

절제는 정보가 흐릿해지는 것이 아니다.
절제는 사용자가 무엇을 봐야 하는지 더 또렷하게 만드는 것이다.

### 8.3 색상

이 문서는 고정 색상표를 제공하지 않는다.
색상은 프로젝트의 브랜드, 산업, 대상 사용자, 콘텐츠 성격에 맞게 정한다.

색상은 역할 기반으로 사용한다.

필요한 색상 역할:

* background.default
* background.subtle
* surface.default
* surface.raised
* surface.selected
* text.primary
* text.secondary
* text.disabled
* border.default
* border.strong
* border.focus
* action.primary
* action.secondary
* action.danger
* status.success
* status.warning
* status.error
* status.info

규칙:

* 한 화면에서 강한 대표 CTA는 하나만 둔다.
* 보조 액션은 약하게 표현한다.
* 상태 색은 실제 상태를 설명할 때만 쓴다.
* 경고색을 마케팅 강조에 쓰지 않는다.
* 성공색을 장식용으로 쓰지 않는다.
* 색상만으로 정보를 구분하지 않는다.
* 브랜드 색상이 접근성 대비를 해치면 조정한다.
* 색보다 문구와 구조로 의미를 전달한다.

### 8.4 타이포그래피

핵심은 특정 폰트가 아니라 읽기 쉬움, 명확한 위계, 짧고 자신감 있는 문장이다.

권장 계층:

| 계층             | 역할               |
| -------------- | ---------------- |
| Display        | 랜딩 페이지 핵심 메시지    |
| Page Title     | 페이지 목적           |
| Section Title  | 섹션 주제            |
| Card Title     | 카드 또는 모듈 제목      |
| Body           | 일반 설명            |
| Body Small     | 보조 설명            |
| Caption        | 메타 정보, 상태, 부연    |
| Legal / Notice | 반드시 알려야 하는 세부 안내 |

규칙:

* 페이지 제목은 화면에서 가장 강한 텍스트여야 한다.
* 본문은 읽기 편한 크기와 줄 간격을 가져야 한다.
* 보조 설명은 작아도 읽을 수 있어야 한다.
* 중요한 정보를 작은 글씨에 숨기지 않는다.
* 버튼 문구는 명확하고 충분히 커야 한다.
* placeholder를 라벨처럼 사용하지 않는다.
* 긴 본문은 너무 넓게 펼치지 않는다.
* 버튼 안 문구는 한 줄로 유지한다.

### 8.5 형태, 표면, 그림자

* 기본 카드는 과한 그림자 없이 구분한다.
* 그림자는 모달, 팝오버, 드롭다운, 떠 있는 패널에만 제한적으로 쓴다.
* 표면 구분은 여백, 배경 차이, 약한 경계로 먼저 해결한다.
* 모든 카드에 그림자를 넣지 않는다.
* 카드, 버튼, 입력창의 모서리 규칙은 일관되게 유지한다.
* 둥근 모서리는 부드러움을 주지만 과하면 장난스러워진다.
* 원형 버튼은 아이콘 버튼처럼 목적이 분명할 때만 쓴다.

### 8.6 아이콘과 그래픽

아이콘과 그래픽은 장식이 아니라 이해를 돕는 도구다.

규칙:

* 아이콘만으로 중요한 정보를 전달하지 않는다.
* 클릭 가능한 아이콘에는 접근 가능한 이름을 제공한다.
* 같은 페이지 안에서 아이콘 스타일을 통일한다.
* 한 섹션에 핵심 그래픽은 하나면 충분하다.
* 그래픽이 제목, 본문, CTA를 방해하면 안 된다.
* 이미지를 정보 전달의 유일한 수단으로 쓰지 않는다.
* 이미지에 담긴 핵심 정보는 텍스트로도 제공한다.
* 사용자의 맥락과 무관한 장식 이미지는 줄인다.

### 8.7 모션

모션은 목적이 분명해야 한다.

허용:

* 버튼 눌림 피드백
* 메뉴 열림
* 모달 등장
* 토스트 등장
* 탭 전환
* 오류 표시
* 카드 hover 강조
* 로딩 진행 상태
* 부드러운 섹션 전환

금지:

* 과한 parallax
* 읽기를 방해하는 자동 애니메이션
* CTA를 흔들거나 깜빡이는 효과
* 불안감을 주는 카운트다운 모션
* 페이지 전환마다 긴 애니메이션
* 스크롤을 강제로 빼앗는 효과
* motion reduction 무시

## 9. 컴포넌트 공통 원칙

컴포넌트는 모양이 아니라 역할, 상태, 문맥, 접근성을 포함해야 한다.

AI는 컴포넌트를 만들 때 다음을 정의한다.

* 이 컴포넌트의 역할은 무엇인가?
* 사용자는 이것을 누르면 무슨 일이 일어난다고 예상하는가?
* 기본 상태는 무엇인가?
* hover, focus, active 상태가 있는가?
* disabled 상태가 있는가?
* loading 상태가 있는가?
* 오류 상태가 있는가?
* 빈 상태가 있는가?
* 접근 가능한 이름이 있는가?
* 키보드로 사용할 수 있는가?
* 좁은 화면에서 어떻게 변하는가?

## 10. 버튼과 CTA

### 10.1 CTA는 행동 문구다

나쁜 CTA:

```text
확인
시작
참여
받기
이동
완료
계속
더보기
```

좋은 CTA:

```text
계정 만들기
문의 보내기
자료 받아보기
일정 선택하기
변경사항 저장하기
입력 내용 확인하기
초대 링크 복사하기
프로젝트 만들기
설정 마치기
파일 업로드하기
```

단, 단계형 흐름 안에서 맥락이 충분히 분명하다면 `다음으로`, `계속하기`를 사용할 수 있다.

### 10.2 CTA 위계

| 위계              | 용도                     |
| --------------- | ---------------------- |
| Primary CTA     | 페이지의 대표 행동             |
| Secondary CTA   | 보조 행동                  |
| Tertiary CTA    | 자세히 보기, 닫기, 뒤로 가기      |
| Neutral CTA     | 취소, 나중에 하기, 돌아가기       |
| Destructive CTA | 삭제, 초기화, 중단처럼 영향이 큰 행동 |

규칙:

* Primary CTA는 한 화면 또는 한 섹션에 하나만 강하게 둔다.
* Secondary CTA는 primary보다 약하게 표현한다.
* 영향이 큰 행동은 일반 CTA와 같은 강도로 두지 않는다.
* 중요한 변경 또는 삭제성 행동은 최종 확인을 둔다.
* CTA가 3개 이상 같은 강도로 보이면 사용자가 헷갈린다.
* 거절 버튼을 흐리게 숨기지 않는다.

### 10.3 CTA 위치

웹사이트에서 CTA는 다음 위치에 둘 수 있다.

* Hero 섹션
* 핵심 설명 직후
* 폼 하단
* 카드 내부
* 비교 영역
* 긴 페이지 중간 반복 지점
* 우측 요약 패널
* sticky 영역
* 페이지 하단

CTA 반복은 목적이 같아야 한다.

좋은 예:

```text
상단 CTA: 시작하기
중간 CTA: 시작하기
하단 CTA: 시작하기
```

나쁜 예:

```text
상단 CTA: 시작하기
중간 CTA: 문의하기
하단 CTA: 자료받기
팝업 CTA: 이벤트 참여하기
```

### 10.4 버튼 상태

버튼은 다음 상태를 가져야 한다.

* 기본
* hover
* focus
* active
* disabled
* loading

규칙:

* loading 중에는 버튼 폭이 갑자기 변하지 않게 한다.
* loading 상태는 중복 클릭을 막아야 한다.
* disabled 상태라면 왜 비활성인지 알 수 있어야 한다.
* 아이콘만 있는 버튼은 접근 가능한 이름이 있어야 한다.

## 11. 폼과 입력

폼은 사용자가 가장 쉽게 불안해하는 영역이다.
따라서 웹 폼은 짧고, 명확하고, 실수 회복이 쉬워야 한다.

### 11.1 폼 원칙

* 한 번에 너무 많은 질문을 하지 않는다.
* 지금 필요한 정보만 묻는다.
* 나중에 물어도 되는 정보는 나중에 묻는다.
* 사용자가 이미 제공한 정보를 다시 묻지 않는다.
* 필수와 선택을 명확히 구분한다.
* 입력 전에 왜 필요한지 알려야 하는 정보는 먼저 알려준다.
* 입력 형식은 사용자가 외우게 하지 않는다.
* 가능한 값은 선택지로 제공한다.
* 오류는 사용자를 탓하지 않는다.
* 제출 후 사용자가 무엇을 기대하면 되는지 알려준다.

### 11.2 라벨

라벨은 placeholder로 대체하지 않는다.

좋은 라벨:

```text
이름
이메일
회사명
팀 이름
프로젝트 이름
문의 내용
희망 일정
초대할 사람
파일 제목
사용 목적
```

나쁜 라벨:

```text
입력
내용
정보
값
항목
데이터
기타
```

### 11.3 도움말

좋은 도움말:

```text
답변을 받을 이메일이에요.
나중에 설정에서 바꿀 수 있어요.
팀원이 알아볼 수 있는 이름으로 입력해 주세요.
업로드한 파일은 이 작업을 처리하는 데만 사용돼요.
선택하지 않아도 다음 단계로 갈 수 있어요.
```

나쁜 도움말:

```text
정확히 입력하세요.
필수값입니다.
올바른 형식으로 입력하세요.
정책상 필요합니다.
```

### 11.4 오류 메시지

오류 메시지는 세 가지를 포함한다.

1. 무슨 일이 일어났는가
2. 왜 그런가
3. 사용자는 무엇을 하면 되는가

좋은 오류:

```text
이메일 형식이 맞지 않아요. name@example.com처럼 입력해 주세요.
파일 용량이 너무 커요. 더 작은 파일로 다시 올려 주세요.
필수 항목이 비어 있어요. 표시된 항목을 입력해 주세요.
연결이 잠시 불안정했어요. 다시 시도해 주세요.
이미 사용 중인 이름이에요. 다른 이름을 입력해 주세요.
```

나쁜 오류:

```text
오류가 발생했습니다.
실패했습니다.
유효하지 않습니다.
다시 시도하세요.
입력값 오류
서버 오류
```

### 11.5 긴 폼

긴 폼은 단계로 나눌 수 있다.

규칙:

* 각 단계의 목적을 제목으로 알려준다.
* 단계 수를 과도하게 늘리지 않는다.
* 현재 위치와 전체 단계를 알려준다.
* 뒤로 가도 입력한 정보가 사라지지 않게 한다.
* 마지막 단계에서 전체 내용을 확인할 수 있게 한다.
* 제출 전에 수정할 수 있어야 한다.
* 완료 후 다음에 무슨 일이 일어나는지 알려준다.

## 12. 카드, 리스트, 표

### 12.1 카드

카드는 정보 묶음이다.
카드를 남발하면 오히려 복잡해진다.

카드가 적합한 경우:

* 서로 다른 정보 단위를 구분할 때
* 여러 선택지를 비교할 때
* 핵심 상태를 요약할 때
* 기능이나 장점을 묶어 보여줄 때
* 대시보드에서 상태를 구분할 때

카드를 쓰지 않아야 하는 경우:

* 단순 텍스트 하나를 강조하려고
* 화면을 예쁘게 채우려고
* 모든 섹션을 박스화하려고
* 카드 안에 또 다른 카드가 반복될 때
* 반복 항목이 너무 많아 리스트나 표가 더 적합할 때

좋은 카드 구조:

```text
카드 제목
짧은 설명
핵심 값 또는 상태
보조 정보
필요한 경우 CTA
```

### 12.2 리스트

리스트는 빠르게 스캔되어야 한다.

기본 구조:

```text
왼쪽: 아이콘, 썸네일, 상태 표시
중앙: 제목 + 짧은 설명
오른쪽: 값, 상태, 이동 표시, 액션
```

규칙:

* 제목은 짧게 쓴다.
* 설명은 필요한 경우에만 쓴다.
* 전체 행이 클릭 가능하면 행 전체가 실제 버튼 또는 링크여야 한다.
* 행 안에 또 다른 버튼을 중첩하지 않는다.
* 이동과 토글을 한 행에 섞지 않는다.
* 반복되는 항목의 구조는 일관되게 유지한다.

### 12.3 표

Simplicity는 표를 없애는 것이 아니라 표를 읽기 쉽게 만드는 것이다.

표가 적합한 경우:

* 여러 항목을 비교해야 할 때
* 숫자, 상태, 날짜, 소유자 등을 함께 봐야 할 때
* 정렬, 필터, 검색이 필요한 데이터가 있을 때
* 반복 데이터가 많을 때
* 관리자 화면에서 항목 관리가 필요할 때

좋은 표는 다음을 갖는다.

* 명확한 열 이름
* 읽기 쉬운 행 높이
* 중요한 값의 시각적 강조
* 상태 텍스트와 상태 표시 병행
* 정렬 가능한 열 표시
* 필터와 검색
* 빈 상태
* 로딩 상태
* 오류 상태
* 접근 가능한 헤더 구조

좁은 화면 대안:

* 카드형 리스트로 전환
* 주요 열만 먼저 표시
* 상세 정보는 펼침
* 가로 스크롤 제공
* 필터를 접힘 영역으로 이동
* 비교가 중요하면 항목 선택 후 비교하도록 구성

## 13. 내비게이션

### 13.1 기본 원칙

좋은 내비게이션:

* 사용자가 예상하는 이름을 쓴다.
* 항목 수가 적다.
* 현재 위치를 알 수 있다.
* 핵심 CTA와 일반 메뉴가 구분된다.
* 페이지 목적을 방해하지 않는다.

나쁜 내비게이션:

* 내부 조직 기준으로 메뉴를 나눈다.
* 추상적 마케팅 용어만 쓴다.
* 모든 링크를 상단에 넣는다.
* CTA가 여러 개다.
* 드롭다운이 너무 깊다.
* 메뉴 이름만 보고 내용을 예상할 수 없다.

### 13.2 메뉴 이름

좋은 메뉴 이름:

```text
기능
활용 사례
가이드
도움말
문의하기
로그인
시작하기
```

나쁜 메뉴 이름:

```text
Experience
Innovation
Platformize
Growth
More
Etc
Resources
Solutioning
```

### 13.3 사이드 내비게이션

긴 문서, 설정, 관리자 화면에서는 사이드 내비게이션을 쓸 수 있다.

규칙:

* 현재 위치를 명확히 표시한다.
* 그룹 이름을 이해하기 쉽게 쓴다.
* 접힘 상태에서도 탐색 가능해야 한다.
* 중요한 액션을 사이드바 아래에 숨기지 않는다.
* 좁은 화면에서는 상단 탭, 목차 버튼, 드롭다운 등으로 전환한다.

## 14. 모달, 팝오버, 패널

오버레이는 사용자의 흐름을 끊는다.
따라서 신중하게 사용한다.

### 14.1 오버레이 사용 전 질문

* 지금 사용자의 작업을 멈춰야 할 만큼 중요한가?
* 페이지 안에서 설명할 수 없는가?
* 사용자가 직접 요청한 정보인가?
* 닫기 쉬운가?
* 키보드와 스크린리더로 사용할 수 있는가?
* 좁은 화면에서도 자연스러운가?
* 사용자를 붙잡기 위한 목적은 아닌가?

이 질문에 답하지 못하면 오버레이를 쓰지 않는다.

### 14.2 모달

적합한 경우:

* 영향이 큰 행동을 확인할 때
* 사용자가 직접 요청한 상세 정보를 보여줄 때
* 짧은 추가 입력이 필요할 때
* 현재 작업을 잠시 멈추고 선택해야 할 때

금지 사례:

* 첫 방문 즉시 가입 강요
* 닫기 어려운 홍보 모달
* 이탈 방지 팝업
* 사용자가 요청하지 않은 제안
* 거절 버튼 없는 안내
* 긴 문서를 작은 모달 안에 밀어 넣기

### 14.3 팝오버

* 트리거 근처에 나타난다.
* 내용은 짧다.
* 중요한 정보는 팝오버 안에만 숨기지 않는다.
* hover에만 의존하지 않는다.
* 키보드와 클릭으로 열고 닫을 수 있다.

### 14.4 사이드 패널

적합한 경우:

* 상세 필터
* 선택 항목 미리보기
* 작업 요약
* 알림 목록
* 문서 목차
* 설정 세부 항목
* 관련 정보 확인

주의:

* 핵심 작업을 패널 안에 숨기지 않는다.
* 패널을 닫아도 작업이 예기치 않게 사라지지 않아야 한다.
* 좁은 화면에서는 전체 페이지 또는 단계형 화면으로 바뀔 수 있다.

## 15. 상태 설계

### 15.1 피드백

사용자의 행동에는 즉각적인 피드백이 있어야 한다.

필요한 피드백:

* 버튼 눌림
* 저장 완료
* 전송 중
* 오류 발생
* 빈 상태
* 성공 결과
* 실패 결과
* 권한 필요
* 연결 불안정
* 처리 대기
* 선택 변경

피드백이 없으면 사용자는 같은 행동을 반복하거나 불안해한다.

### 15.2 로딩

규칙:

* 가능한 한 로딩을 줄인다.
* 구조를 알 수 있는 곳에는 skeleton을 사용한다.
* 짧은 로딩에는 버튼 내부 로딩을 사용한다.
* 긴 로딩에는 설명을 제공한다.
* 오래 걸릴 수 있으면 사용자가 무엇을 기대하면 되는지 알려준다.
* 무한 스피너만 보여주지 않는다.
* 로딩 실패 시 오류 상태로 전환한다.

좋은 문구:

```text
내용을 불러오고 있어요.
파일을 확인하고 있어요.
작업을 준비하고 있어요.
잠시 후 다시 이어서 할 수 있어요.
```

나쁜 문구:

```text
Loading...
처리중...
잠시만 기다려주세요.
```

### 15.3 빈 상태

빈 상태는 사용자가 다음 행동을 이해하게 하는 기회다.

좋은 빈 상태:

```text
아직 만든 프로젝트가 없어요.
첫 프로젝트를 만들고 팀과 함께 시작해 보세요.
[프로젝트 만들기]
```

검색 결과 없음:

```text
조건에 맞는 결과가 없어요.
검색어를 바꾸거나 필터를 줄여 보세요.
[필터 초기화하기]
```

나쁜 빈 상태:

```text
데이터가 없습니다.
```

### 15.4 성공 상태

성공 상태는 사용자가 안심하게 해야 한다.

포함할 것:

* 무엇이 완료됐는지
* 다음에 무엇을 기대하면 되는지
* 사용자가 이어서 할 수 있는 행동

예:

```text
요청을 보냈어요.
답변이 오면 알림 영역에서 확인할 수 있어요.
[요청 내역 보기]
```

### 15.5 실패 상태

실패 상태는 사용자를 막다른 길에 두면 안 된다.

포함할 것:

* 어떤 작업이 실패했는지
* 가능한 이유
* 해결 방법
* 대안

예:

```text
파일을 올리지 못했어요.
파일 형식이나 용량을 확인한 뒤 다시 올려 주세요.
[다시 업로드하기]
```

### 15.6 토스트

토스트는 짧은 상태 알림이다.

적합한 경우:

```text
저장했어요.
복사했어요.
초대를 보냈어요.
설정을 바꿨어요.
파일을 올렸어요.
```

부적합한 경우:

* 긴 오류 설명
* 중요한 확인
* 긴 안내
* 사용자가 반드시 읽어야 하는 내용
* 영향이 큰 선택

규칙:

* 한 문장으로 쓴다.
* CTA나 입력을 가리지 않는다.
* 같은 토스트를 반복해서 띄우지 않는다.
* 스크린리더가 인지할 수 있게 한다.

## 16. UX Writing

### 16.1 기본 문체

권장:

* 짧은 문장
* 능동형
* 긍정형
* 일상어
* 부드러운 존댓말
* 강요보다 안내
* 사용자의 상황을 이해하는 표현

피해야 할 것:

* 과한 경어
* 딱딱한 행정 문구
* 명사형 나열
* 내부 용어
* 기술 용어 남발
* 불안 자극
* 죄책감 유도
* 과한 유머
* 모호한 마케팅 표현

### 16.2 짧게 쓴다

나쁜 예:

```text
서비스 이용을 원활하게 진행하기 위해서는 아래 정보를 모두 입력해 주셔야 합니다.
```

좋은 예:

```text
시작하려면 아래 정보가 필요해요.
```

### 16.3 능동형으로 쓴다

나쁜 예:

```text
요청이 처리되었습니다.
```

좋은 예:

```text
요청을 처리했어요.
```

### 16.4 긍정형으로 쓴다

나쁜 예:

```text
이름을 입력하지 않으면 다음 단계로 이동할 수 없어요.
```

좋은 예:

```text
이름을 입력하면 다음 단계로 갈 수 있어요.
```

단, 위험하거나 되돌릴 수 없는 행동은 명확하게 말한다.

```text
삭제하면 다시 복구할 수 없어요.
```

### 16.5 제목 문구

제목은 페이지의 목적을 알려야 한다.

나쁜 제목:

```text
Welcome
새로운 경험을 시작하세요
더 나은 미래를 위한 혁신
```

좋은 제목:

```text
팀 작업을 한 곳에서 정리하세요
프로젝트 이름을 정해 주세요
문의 내용을 알려 주세요
초대할 사람을 선택해 주세요
작업을 시작하기 전에 설정을 확인해요
```

### 16.6 설명 문구

설명은 제목을 반복하지 않는다.

나쁜 예:

```text
문의 내용을 입력해 주세요.
아래에 문의 내용을 입력할 수 있어요.
```

좋은 예:

```text
상황을 자세히 적어 주시면 더 정확히 도와드릴 수 있어요.
```

### 16.7 버튼 문구 패턴

| 상황    | 좋은 CTA     |
| ----- | ---------- |
| 계정 생성 | 계정 만들기     |
| 문의    | 문의 보내기     |
| 자료 요청 | 자료 받아보기    |
| 일정 선택 | 일정 선택하기    |
| 저장    | 변경사항 저장하기  |
| 업로드   | 파일 업로드하기   |
| 초대    | 초대 보내기     |
| 검색    | 결과 보기      |
| 설정    | 설정 마치기     |
| 확인    | 입력 내용 확인하기 |
| 재시도   | 다시 시도하기    |
| 취소    | 취소하기       |
| 삭제    | 삭제하기       |

### 16.8 금지 문구

* 사용자를 겁주는 문구를 남용하지 않는다.
* 거절 버튼에 “혜택 포기하기”처럼 압박성 문구를 쓰지 않는다.
* 취소 행동에 죄책감을 유발하지 않는다.
* “마지막 기회” 같은 거짓 긴급성을 쓰지 않는다.
* “모두가 선택한” 같은 검증 불가능한 표현을 남용하지 않는다.
* “무료”라고 쓰면서 조건을 숨기지 않는다.
* “간단히”라고 쓰면서 긴 절차를 요구하지 않는다.
* “필수”가 아닌 항목을 필수처럼 표현하지 않는다.
* “동의”의 범위를 모호하게 쓰지 않는다.

## 17. 다크패턴 방지

사용자를 속이거나 몰아붙이는 디자인은 금지한다.
전환이 높아도 사용자가 속았다고 느끼면 실패다.

### 17.1 금지되는 패턴

| 패턴        | 설명                       | 대안                   |
| --------- | ------------------------ | -------------------- |
| 강제 팝업     | 첫 진입 즉시 특정 행동을 강요        | 사용자가 맥락을 이해한 뒤 제안    |
| 숨겨진 닫기    | 닫기 버튼을 작게 숨김             | 명확한 닫기 제공            |
| 이탈 방해     | 나가려는 사용자를 반복해서 붙잡음       | 저장되지 않은 변경이 있을 때만 확인 |
| 모호한 CTA   | 버튼만 보고 다음 행동을 알 수 없음     | 행동 중심 라벨 사용          |
| 불안 유도     | 손해, 마감, 후회를 과장           | 실제 조건만 중립적으로 설명      |
| 죄책감 유도    | 거절 버튼에 부정적 문구 사용         | 중립적인 거절 문구           |
| 숨겨진 조건    | 중요한 조건을 뒤늦게 공개           | 결정 전에 핵심 조건 공개       |
| 사전 선택     | 원치 않을 수 있는 항목을 기본 선택     | 사용자가 직접 선택           |
| 복잡한 중단 경로 | 시작보다 중단이 훨씬 어려움          | 중단 경로 명확히 제공         |
| 가짜 긴급성    | 실제 제한이 없는데 급한 것처럼 표시     | 실제 제한만 표시            |
| 광고 위장     | 광고·추천·제휴를 일반 콘텐츠처럼 보이게 함 | 성격을 명확히 표시           |
| 선택권 불균형   | 원하는 선택만 강하고 거절은 흐림       | 선택권을 공정하게 제공         |
| 과도한 정보 요청 | 필요 이상 정보를 요구             | 필요한 이유와 범위 설명        |

### 17.2 선택지 공정성

* 동의와 거절은 모두 이해 가능해야 한다.
* 거절 버튼을 숨기지 않는다.
* 수락은 한 번, 거절은 여러 번 클릭하게 만들지 않는다.
* 기본 선택은 사용자의 기대와 이익에 맞아야 한다.
* 사용자가 선택하지 않았는데 선택한 것처럼 처리하지 않는다.
* 선택을 바꾸는 방법을 제공한다.
* 한 선택지는 긍정적으로, 다른 선택지는 부정적으로 몰아가지 않는다.

### 17.3 비용, 조건, 제한

* 비용, 제한, 조건은 결정 전에 보여준다.
* 추가 비용은 마지막 단계에서 갑자기 등장하지 않아야 한다.
* 체험, 할인, 무료 표현에는 조건을 함께 보여준다.
* 취소 조건이 있으면 시작 전에 알려준다.
* 사용량 제한이 있으면 선택 전에 알려준다.
* 자동 결제나 자동 갱신은 명확한 동의를 받아야 한다.

### 17.4 권한 요청

알림, 위치, 카메라, 마이크, 파일 접근 등 권한은 다음 순서로 요청한다.

1. 먼저 사용자가 가치를 이해하게 한다.
2. 왜 필요한지 설명한다.
3. 사용자가 예상할 수 있는 시점에 요청한다.
4. 거절해도 가능한 대안을 제공한다.
5. 반복 요청하지 않는다.

좋은 예:

```text
파일을 첨부하려면 기기에서 파일을 선택해야 해요.
[파일 선택하기]
```

나쁜 예:

```text
첫 진입 즉시 브라우저 권한 요청
```

### 17.5 중단, 삭제, 해지

* 중단, 삭제, 해지 경로를 숨기지 않는다.
* 시작보다 중단이 과도하게 어려우면 안 된다.
* 계속 묻지 않는다.
* 영향은 사실만 설명한다.
* 되돌릴 수 있는지 명확히 알려준다.
* 되돌릴 수 없는 경우 최종 확인을 둔다.

좋은 예:

```text
삭제하면 다시 복구할 수 없어요.
그래도 삭제할까요?
[삭제하기] [취소]
```

나쁜 예:

```text
정말 포기하시겠어요?
이 선택을 하면 큰 손해를 볼 수 있어요.
```

### 17.6 추천, 광고, 제휴성 콘텐츠

* 추천인지 광고인지 명확히 표시한다.
* 자동 정렬 결과라면 기준을 설명할 수 있어야 한다.
* 광고를 일반 기능 버튼처럼 보이게 하지 않는다.
* 사용자가 원하지 않으면 닫거나 줄일 수 있어야 한다.
* 거절 후 같은 제안을 반복해서 강요하지 않는다.

### 17.7 개인정보와 데이터

* 어떤 데이터를 왜 요청하는지 설명한다.
* 필요 이상의 데이터를 요구하지 않는다.
* 데이터 사용 범위를 모호하게 쓰지 않는다.
* 선택 데이터 제공을 필수처럼 만들지 않는다.
* 데이터 공유 대상이 있다면 알려준다.
* 공개 범위가 있는 콘텐츠는 현재 공개 상태를 명확히 보여준다.
* 사용자가 공개 범위를 변경할 수 있어야 한다.
* 데이터 삭제 방법을 제공한다.

## 18. 접근성

접근성은 별도 옵션이 아니라 기본 품질이다.
Simplicity는 보조기술 사용자에게도 이해 가능해야 한다.

### 18.1 기본 원칙

* 실제 버튼은 `<button>`을 사용한다.
* 이동은 `<a>`를 사용한다.
* 폼은 `<form>`으로 묶는다.
* 라벨은 입력과 연결한다.
* 모든 인터랙티브 요소에는 접근 가능한 이름이 있어야 한다.
* 키보드만으로 모든 기능을 사용할 수 있어야 한다.
* 포커스 위치가 항상 보여야 한다.
* 색상만으로 상태를 전달하지 않는다.
* 이미지와 아이콘에는 적절한 대체 텍스트를 제공한다.
* 모달은 포커스 이동과 닫기 동작을 지원한다.
* 오류는 스크린리더가 인지할 수 있어야 한다.
* 사용자가 모션 감소를 선호하면 애니메이션을 줄인다.

### 18.2 금지 구현

```text
div를 버튼처럼 사용
span을 링크처럼 사용
button 안에 button 넣기
클릭 가능한 카드 안에 또 다른 클릭 요소 중첩
hover에만 중요한 정보 표시
placeholder를 label로 사용
색만으로 오류 표시
키보드 포커스 제거
outline: none만 적용
테이블 행 전체에 onClick만 붙이기
```

### 18.3 포커스

* 포커스 링은 명확히 보인다.
* 포커스가 헤더나 sticky 영역에 가려지지 않는다.
* 모달이 열리면 모달 안으로 포커스가 이동한다.
* 모달이 닫히면 원래 트리거로 포커스가 돌아간다.
* 탭 순서는 화면의 시각적 순서와 일치한다.
* 긴 페이지에는 본문으로 건너뛰기 링크를 제공할 수 있다.

### 18.4 대체 텍스트

예:

```text
닫기 아이콘 → aria-label="닫기"
검색 아이콘 → aria-label="검색"
성공 장식 이미지 → alt=""
상태 그래프 → 핵심 추세를 텍스트로 설명
```

원칙:

* 의미 있는 이미지는 의미를 설명한다.
* 장식 이미지는 빈 alt를 사용한다.
* 아이콘 버튼은 기능을 설명한다.
* 그래프나 차트는 핵심 내용을 텍스트로도 제공한다.

### 18.5 폼 접근성

* 모든 입력 필드에는 라벨이 있어야 한다.
* 오류 메시지는 필드와 연결되어야 한다.
* 필수 입력은 명확히 표시한다.
* 입력 형식 힌트는 필드 근처에 둔다.
* 라디오와 체크박스 그룹은 질문 문구와 연결한다.
* 제출 실패 시 오류 위치를 쉽게 찾을 수 있어야 한다.

## 19. 신뢰와 투명성

범용 웹사이트에서 신뢰를 만들려면 사용자가 결정 전에 알아야 할 정보를 명확히 보여줘야 한다.

### 19.1 결정 전에 알려야 할 정보

* 이 서비스나 기능이 무엇을 하는지
* 사용자가 무엇을 제공해야 하는지
* 제공한 정보가 어떤 목적으로 쓰이는지
* 사용자가 어떤 결과를 기대할 수 있는지
* 결과가 언제 또는 어떤 조건에서 달라질 수 있는지
* 사용자가 선택을 바꾸거나 중단할 수 있는지
* 사용자가 실수했을 때 어떻게 회복할 수 있는지
* 제한, 예외, 주의사항이 있는지
* 자동으로 일어나는 동작이 있는지
* 외부 서비스, 제휴, 추천, 광고성 요소가 포함되는지
* 문의하거나 도움을 받을 수 있는 경로가 있는지
* 데이터가 저장, 수정, 삭제되는 방식이 사용자의 기대와 일치하는지

### 19.2 신뢰 정보 배치

* 사용자의 결정에 직접 영향을 주는 정보는 CTA 근처에 둔다.
* 자세한 정보는 접힘 영역, FAQ, 도움말, 문서로 제공한다.
* 민감한 정보를 요청하기 전에는 이유를 알려준다.
* 영향이 큰 행동 전에는 결과와 되돌릴 수 있는지 알려준다.
* 자동 동작이 있다면 사전에 알려준다.
* 사용자가 궁금해할 만한 정보는 찾기 쉬운 위치에 둔다.

### 19.3 민감한 행동

민감한 행동에는 더 명확한 확인이 필요하다.

* 공개 범위 변경
* 데이터 삭제
* 권한 부여
* 비용 발생
* 다른 사용자에게 영향을 주는 행동
* 되돌릴 수 없는 변경
* 외부 공유
* 계정, 조직, 프로젝트 단위 변경

민감한 행동의 버튼은 일반 행동과 구분하고, 실수로 누르기 쉬운 위치에 두지 않는다.

## 20. 검색, 필터, 페이지네이션

### 20.1 검색

* 검색이 중요한 서비스에서는 검색창을 찾기 쉽게 둔다.
* placeholder는 검색 가능한 대상을 설명한다.
* 검색 결과는 검색어와 함께 보여준다.
* 결과가 없을 때는 다른 검색어를 제안한다.
* 자동완성은 사용자를 방해하지 않아야 한다.
* 최근 검색어는 삭제할 수 있어야 한다.
* 검색 결과 정렬 기준을 알 수 있어야 한다.
* 검색 입력은 Enter와 버튼 클릭 모두 지원한다.

### 20.2 필터

* 필터가 적용되면 적용된 조건을 명확히 보여준다.
* 필터는 쉽게 해제할 수 있어야 한다.
* 조건이 많으면 그룹화한다.
* 가장 많이 쓰는 필터를 먼저 배치한다.
* 고급 필터는 접어둘 수 있다.
* 필터 결과 수를 가능한 경우 표시한다.
* 필터 적용 방식은 즉시 적용 또는 적용 버튼 중 하나로 일관한다.
* 필터가 결과를 모두 없애면 해결 방법을 제안한다.

### 20.3 페이지네이션과 더보기

* 사용자가 특정 위치를 기억해야 하면 페이지네이션이 적합하다.
* 콘텐츠 탐색이 목적이면 더보기 또는 무한 스크롤을 사용할 수 있다.
* 업무형 데이터에는 페이지네이션이 더 예측 가능하다.
* 무한 스크롤에서는 푸터 접근성을 고려한다.
* 현재 페이지와 전체 페이지 정보를 제공한다.
* 필터 변경 시 페이지를 초기화할 수 있다.
* 키보드 사용자도 페이지 이동을 할 수 있어야 한다.

## 21. 데이터와 시각화

### 21.1 숫자 표현

* 숫자는 단위와 함께 표시한다.
* 기간이 중요한 숫자는 기간을 함께 표시한다.
* 비교가 필요한 숫자는 기준을 함께 표시한다.
* 변화율은 상승과 하락의 의미를 명확히 한다.
* 큰 숫자는 읽기 쉽게 포맷한다.
* 예상값과 확정값을 구분한다.
* 실시간 값은 업데이트 시점을 함께 표시할 수 있다.
* 숫자만 크게 보여주고 의미를 설명하지 않는다.

### 21.2 차트

* 차트는 질문에 답해야 한다.
* 차트 제목은 무엇을 보여주는지 말해야 한다.
* 축, 단위, 기간을 명확히 표시한다.
* 색상만으로 데이터를 구분하지 않는다.
* 범례는 이해하기 쉬워야 한다.
* 선이 너무 많으면 차트를 나눈다.
* 원형 차트는 비교가 어려우므로 신중히 사용한다.
* 차트 아래에 핵심 해석을 짧게 제공할 수 있다.
* 차트 로딩, 빈 상태, 오류 상태를 제공한다.
* 데이터 출처나 기준이 필요하면 표시한다.

### 21.3 상태 지표

* 상태는 명확한 이름을 가져야 한다.
* 상태 색상은 보조 수단이다.
* 상태 라벨은 혼동되지 않아야 한다.
* 다음 상태를 예측할 수 있어야 한다.
* 문제가 있는 상태는 해결 방법을 제공한다.
* 상태 체계는 서비스 전체에서 일관되어야 한다.

## 22. 페이지 유형별 설계 레시피

### 22.1 랜딩 페이지

목적:

* 사용자가 서비스 가치를 빠르게 이해하고 다음 행동을 결정하게 한다.

구조:

```text
Header
Hero
  - 명확한 가치 제안
  - 짧은 설명
  - 대표 CTA
  - 보조 링크
Problem
Solution
Core Benefits
How It Works
Use Cases
Trust Information
FAQ
Final CTA
Footer
```

규칙:

* Hero에서 너무 많은 기능을 말하지 않는다.
* 첫 CTA는 하나만 강하게 둔다.
* 추상적인 문구보다 사용자가 얻는 변화를 말한다.
* 문제를 먼저 말하고 기능은 그다음 설명한다.
* CTA를 반복하되 목적은 일관되게 유지한다.
* 신뢰 요소는 과장하지 않는다.
* FAQ는 전환 방해가 아니라 불안 해소용으로 둔다.

### 22.2 소개 페이지

목적:

* 사용자가 제품, 서비스, 조직, 프로젝트의 성격을 이해하게 한다.

구조:

```text
Page Title
요약 설명
핵심 가치
주요 기능 또는 특징
사용 흐름
대상 사용자
신뢰 정보
다음 행동
```

규칙:

* 내부 관점보다 사용자 관점으로 설명한다.
* “우리는 무엇을 만들었는가”보다 “사용자는 무엇을 할 수 있는가”를 먼저 말한다.
* 추상어보다 구체적인 사용 장면을 보여준다.
* CTA는 소개 흐름을 방해하지 않게 둔다.

### 22.3 제품 또는 기능 상세 페이지

목적:

* 사용자가 특정 대상의 가치, 조건, 사용 방법을 판단하게 한다.

구조:

```text
Title
핵심 요약
주요 이점
사용 장면
작동 방식
조건 또는 제한
비교 정보
FAQ
CTA
```

규칙:

* 이름보다 사용자가 해결할 문제를 먼저 보여준다.
* 주요 이점은 3개 내외로 압축한다.
* 조건, 제한, 취소, 변경 가능 여부를 숨기지 않는다.
* 이미지나 데모는 실제 사용 맥락을 보여줘야 한다.
* 페이지 하단에는 자연스러운 다음 행동을 제공한다.

### 22.4 시작하기 / 등록 흐름

목적:

* 사용자가 실수 없이 첫 설정이나 등록을 마치게 한다.

구조:

```text
Minimal Header
Step Title
Short Explanation
Form or Selection
Inline Help
Inline Error
Primary CTA
Optional Summary
```

규칙:

* 한 단계에 한 가지 질문 묶음만 보여준다.
* 왜 이 정보를 입력해야 하는지 알려준다.
* 뒤로 가도 입력값을 유지한다.
* 마지막에는 입력 내용을 확인할 수 있게 한다.
* 선택 항목은 기본 선택하지 않는다.
* 완료 후 다음에 무엇을 하면 되는지 알려준다.

### 22.5 문의 / 요청 페이지

목적:

* 사용자가 필요한 정보를 쉽게 전달하고 이후 흐름을 이해하게 한다.

구조:

```text

짧은 안내

입력 폼
첨부 또는 추가 정보

```

규칙:

* 어떤 정보를 적으면 좋은지 안내한다.
* 입력해야 할 항목을 최소화한다.
* 사용자가 응답 방식이나 이후 흐름을 예측할 수 있게 한다.
* 제출 후 완료 상태를 명확히 보여준다.
* 오류가 나면 작성한 내용을 잃지 않게 한다.

### 22.6 옵션 비교 / 가격 페이지

목적:

* 사용자가 여러 선택지 중 자신에게 맞는 것을 고르게 한다.

구조:


규칙:

* 추천 옵션을 강조할 수 있지만 다른 선택지를 흐리게 속이면 안 된다.
* 선택 기준을 명확히 설명한다.
* 제한이나 차이를 숨기지 않는다.
* 사용자가 비교해야 할 항목만 표에 남긴다.
* 모든 옵션의 CTA를 같은 강도로 만들지 않는다.
* 사용자가 선택을 바꿀 수 있는지 알려준다.

### 22.7 대시보드

목적:

* 사용자가 현재 상태와 다음 행동을 빠르게 파악하게 한다.

구조:

```text

Key Metrics or Summary

Recent Activity
```

규칙:

* 가장 중요한 상태를 첫 카드에 둔다.
* 숫자보다 해석이 중요하면 해석을 함께 제공한다.
* 모든 카드를 같은 강도로 만들지 않는다.
* 알림은 심각도에 따라 구분한다.
* 빈 상태와 오류 상태를 반드시 설계한다.
* 반복 사용 화면이므로 과한 배너와 장식을 줄인다.

### 22.8 설정 페이지

목적:

* 사용자가 현재 설정을 이해하고 안전하게 변경하게 한다.


규칙:

* 설정 그룹 이름을 사용자 언어로 쓴다.
* 토글과 이동 행을 혼동하지 않게 한다.
* 변경 즉시 적용인지 저장이 필요한지 명확히 한다.
* 영향이 큰 설정은 별도 영역에 둔다.
* 삭제, 초기화, 중단은 별도 확인을 둔다.
* 현재 상태를 명확히 표시한다.

### 22.9 도움말 / FAQ 페이지

목적:

* 사용자가 문제 해결 방법을 빠르게 찾게 한다.

```text

Status or Notice
```

규칙:

* 검색을 상단에 둔다.
* 자주 찾는 문제를 먼저 보여준다.
* 카테고리 이름은 사용자 언어로 쓴다.
* 답변은 결론부터 쓴다.
* 문의 경로를 숨기지 않는다.
* 자동화된 안내만 강요하지 않는다.
* 해결이 안 될 경우 다음 경로를 제공한다.

### 22.10 문서 / 아티클 페이지

목적:

* 사용자가 긴 글을 편하게 읽고 필요한 행동으로 이어지게 한다.

* 목차는 긴 글에만 둔다.
* 문단을 짧게 유지한다.
* 이미지에는 설명을 제공한다.
* 중간 CTA는 글 흐름을 방해하지 않아야 한다.
* 구독, 추천, 홍보 요소로 독서를 막지 않는다.

### 22.11 목록형 페이지

목적:

규칙:

* 핵심 정보는 먼저 보여준다.
* 보조 정보는 약하게 보여준다.
* 항목별 주요 행동은 일관된 위치에 둔다.
* 목록이 길면 검색, 필터, 정렬을 제공한다.
* 빈 목록은 다음 행동을 제안한다.

* 위험 행동은 하단이나 별도 영역에 둔다.
* 편집, 삭제, 공유 같은 행동은 역할에 맞게 계층화한다.
* 목록으로 돌아갈 수 있어야 한다.

## 24. 개발 구현 가이드

### 24.1 HTML

* 가능한 한 기본 HTML 요소를 사용한다.
* 버튼은 `button` 요소를 사용한다.
* 이동은 `a` 요소를 사용한다.
* 폼은 `form` 구조를 사용한다.
* 라벨은 input과 연결한다.
* 목록은 `ul`, `ol`, `li`를 사용한다.
* 데이터 표는 `table`을 사용한다.
* 문서 구조는 `header`, `main`, `nav`, `section`, `footer`를 사용한다.
* 커스텀 컴포넌트는 접근성 속성을 보완한다.
* 의미 없는 div 중첩을 줄인다.

### 24.2 CSS

* 색상은 역할 토큰으로 관리한다.
* 간격은 스케일 토큰으로 관리한다.
* 폰트 크기와 줄 간격은 타입 토큰으로 관리한다.
* 컴포넌트 스타일은 상태별로 정의한다.
* hover, focus, active, disabled 상태를 분리한다.
* 반응형 규칙은 콘텐츠 구조에 맞춰 작성한다.
* 페이지 레이아웃과 컴포넌트 스타일을 분리한다.
* 다크 모드를 지원할 경우 역할 토큰을 재정의한다.
* 하드코딩된 일회성 스타일을 줄인다.

### 24.3 JavaScript

* 클릭 후 즉시 상태를 업데이트한다.
* 비동기 요청에는 로딩과 오류 상태를 제공한다.
* 폼 제출 중 중복 제출을 방지한다.
* 입력 내용은 가능한 경우 보존한다.
* 모달 포커스 관리를 구현한다.
* 드롭다운 키보드 조작을 지원한다.
* URL 상태가 필요한 필터와 검색은 URL에 반영한다.
* 사용자가 뒤로 가기를 눌렀을 때 예상 가능한 상태로 돌아간다.
* JavaScript 실패 시 핵심 콘텐츠 접근을 최대한 유지한다.

### 24.4 접근성 구현

* aria는 기본 HTML로 해결할 수 없을 때 보조적으로 사용한다.
* aria-label은 보이는 라벨을 대체하지 않는 것이 좋다.
* aria-live는 상태 변화 알림에 신중히 사용한다.
* role을 잘못 사용해 기본 의미를 깨지 않는다.
* 포커스 트랩은 모달에서만 적절히 사용한다.
* skip link를 고려한다.
* form error와 input을 연결한다.
* disabled와 aria-disabled의 차이를 이해하고 사용한다.
* 접근성 검사는 자동 도구와 수동 키보드 테스트를 함께 한다.

```text
- 오류 메시지는 필드 아래에 원인과 해결 방법을 함께 제공한다.
- 제출 버튼 문구는 행동 결과를 말한다.
- 취소나 뒤로 가기 행동을 숨기지 않는다.
- 저장되지 않은 변경 사항을 보호한다.
- 완료 후 사용자가 결과를 확인할 수 있게 한다.
```

### 25.5 설정 페이지 생성 프롬프트

```text
이 DESIGN.md를 기준으로 설정 페이지를 만든다.

목표:
- 사용자가 통제감을 느껴야 한다.
- 설정 항목은 사용자의 목적에 맞게 그룹화한다.
- 각 설정의 현재 상태를 명확히 보여준다.
- 즉시 적용과 저장 필요 설정을 구분한다.
- 개인정보, 공개 범위, 알림, 권한 설정은 특히 명확하게 쓴다.
- 위험한 행동은 별도 영역에 배치한다.
- 해지, 삭제, 비활성화 경로를 숨기지 않는다.
- 설정 변경 결과를 피드백으로 알려준다.
- 어려운 용어는 쉬운 설명을 붙인다.
- 광고나 업셀을 설정 흐름에 과하게 섞지 않는다.
```

### 25.6 콘텐츠 페이지 생성 프롬프트

```text

- 제목은 내용의 핵심을 정확히 말한다.
- 서론은 독자가 얻을 내용을 짧게 알려준다.
- 본문은 짧은 문단과 명확한 중간 제목으로 구성한다.
- 긴 콘텐츠에는 목차를 제공한다.
- 본문 너비는 읽기 편한 범위로 제한한다.
- 이미지, 표, 인용은 이해를 돕는 경우에만 사용한다.
- CTA는 본문 흐름을 방해하지 않게 배치한다.
- 관련 콘텐츠는 본문 후반에 제공한다.
- 광고나 추천 요소가 읽기 흐름을 끊지 않게 한다.
```

## 27. 반례와 수정 방향

### 27.1 버튼이 너무 많음

문제:

* 사용자가 무엇을 눌러야 할지 모른다.

수정:

* 주 행동 하나를 정한다.
* 보조 행동은 링크나 보조 버튼으로 낮춘다.
* 위험 행동은 별도 영역으로 분리한다.
* 불필요한 CTA를 제거한다.

### 27.2 모든 정보가 카드 안에 있음

문제:

* 화면이 조각나고 비교가 어렵다.

수정:

* 반복 데이터는 리스트나 표로 바꾼다.
* 핵심 카드만 남긴다.
* 관련 정보를 그룹화한다.
* 카드 내부 구조를 단순화한다.

### 27.3 설명이 추상적임

문제:

* 사용자가 가치를 이해하지 못한다.

수정:

* 사용자의 상황을 말한다.
* 결과를 구체적으로 말한다.
* 기능명이 아니라 해결되는 문제를 말한다.
* 예시를 짧게 제공한다.

### 27.4 거절이 어려움

문제:

* 사용자가 조작당한다고 느낀다.

수정:

* 거절 버튼을 명확히 제공한다.
* 거절 문구를 중립적으로 쓴다.
* 수락과 거절의 단계 수를 비슷하게 맞춘다.
* 나중에 변경할 수 있음을 알려준다.

### 27.5 모바일 앱 같은 데스크톱

문제:

* 넓은 화면을 낭비한다.

수정:

* 컨테이너 최대 너비를 조정한다.
* 보조 정보를 옆 패널로 배치한다.
* 목록과 상세를 함께 보여준다.
* 상단 또는 사이드 내비게이션을 사용한다.

### 27.6 색상에만 의존

문제:

* 색상을 구분하지 못하면 상태를 이해할 수 없다.

수정:

* 텍스트 라벨을 추가한다.
* 아이콘이나 형태를 함께 사용한다.
* 상태 메시지를 제공한다.
* 대비를 높인다.

### 27.7 오류가 불친절함

문제:

* 사용자가 해결 방법을 모른다.

수정:

* 원인을 설명한다.
* 해결 행동을 제공한다.
* 해당 필드 가까이에 표시한다.
* 입력값을 보존한다.

### 27.8 Hero가 과장됨

문제:

* 신뢰보다 광고처럼 보인다.

수정:

* 구체적인 가치 제안으로 바꾼다.
* 대상 사용자를 명확히 한다.
* 보조 문구를 실제 사용 상황으로 쓴다.
* CTA를 하나로 정리한다.

## 28. 최종 원칙

이 문서를 따르는 AI는 다음을 최종 기준으로 삼는다.

1. 사용자가 이해하지 못하면 실패다.
2. 사용자가 예측하지 못하면 실패다.
3. 사용자가 거절할 수 없으면 실패다.
4. 사용자가 중요한 조건을 나중에 알게 되면 실패다.
5. 사용자가 오류 후 무엇을 해야 할지 모르면 실패다.
6. 사용자가 색상이나 장식 때문에 핵심 행동을 놓치면 실패다.
7. 특정 브랜드의 외형을 베끼면 실패다.
8. 특정 색상을 쓰지 않아도 사용자가 편하고 명확하게 끝낼 수 있다면 성공이다.
9. 복잡함을 사용자에게 떠넘기지 않아야 한다.
10. 사용자의 선택권, 시간, 신뢰를 존중해야 한다.

최종 목표는 단순히 미니멀한 웹사이트가 아니다.

최종 목표는 다음이다.

> 사용자가 복잡한 일을 단순하게 느끼고, 불안 없이 이해하고, 실수 없이 행동을 끝낼 수 있는 웹사이트.
