# 12-Factor Agent 원칙으로 본 드리프트리스

HumanLayer의 [12-Factor Agents](https://github.com/humanlayer/12-factor-agents)는
실제 서비스에 투입해도 버티는 LLM 에이전트를 만들기 위한, 널리 인정받는 점검
목록이에요. 라이브러리가 아니라 *설계* 기준 문서예요. "믿을 만한 에이전트라면 어떤
성질을 갖춰야 하는가"를 말해 주고, 그 성질을 어떻게 갖출지는 각 프로젝트에
맡깁니다.

이 문서는 "참고 자료를 읽고 도입했다"는 말의 정직한 버전이에요. 12요소의 구조나
코드를 **베끼지 않습니다.** 대신 한 요소씩 이렇게 물어요. *드리프트리스는 이미 이걸
체화하고 있나, 그렇다면 어떻게? 아니면 아직 더 잘해야 할 부분인가?* "이미 한다"면
실제 장치를 가리키고, "부분적"이면 솔직히 그렇다고 말하고 빠진 부분의 이름을
댑니다. 완성도가 좋아 보이려고 초록불을 켜는 요소는 하나도 없어요.

> 한 줄 요약: **드리프트리스는 "프롬프트 소유", "컨텍스트 소유", "작고 집중된
> 에이전트", "사람을 게이트 도구로 호출하기" 요소를 강하게 체화하고 있어요. 반면
> "명시적 제어 흐름", "상태 통합", "일시정지/재개" 요소는 부분적이에요. 이건 우리
> 코드보다 하네스(실행 틀)에 더 기대고 있는데, 그걸 숨기지 않고 그대로 말합니다.**

"프롬프트/컨텍스트 소유", "작고 집중된 에이전트", "사람 호출", "상태 통합", "에러
압축" 같은 핵심 아이디어가 어디에 떨어지는지 한눈에 보면 이래요.

| 12요소 원칙 | 드리프트리스 판정 | 어디에 있나 |
|---|---|---|
| 2. 프롬프트를 직접 소유 | **잘 체화함** | 단일 원본 `profiles/shared/` + 거울 일치 게이트 |
| 3. 컨텍스트 창을 직접 소유 | **잘 체화함** | 가벼운 핵심 규칙, 토큰 폭주 가드, 증거 파일 분리 |
| 7. 사람을 도구 호출로 부르기 | **잘 체화함** | 관리자 전용 4개 게이트, 쉬운 말 보고서 |
| 10. 작고 집중된 에이전트 | **잘 체화함** | 가벼운 스킬, 부모 아래 좁은 밤샘 워커 |
| 9. 에러를 컨텍스트로 압축 | **부분적** | 짧은 UNVERIFIED 상태 + 1회 재시도 규칙. 구조적 에러요약 루프는 아직 |
| 5. 실행 상태와 업무 상태 통합 | **부분적** | 결정 등록부 + 이슈/PR 본문이 상태. 단일 기계검증 저장소는 아님 |
| 8. 제어 흐름을 직접 소유 | **부분적** | 게이트 + 사다리가 흐름을 제약. 단계 오케스트레이션은 하네스 몫 |
| 6. 실행/일시정지/재개 | **부분적** | 런처 래퍼로 재개. 정식 체크포인트 API는 없음 |
| 12. 상태 없는 리듀서 | **부분적(비유로)** | 매 세션 의도를 다시 읽음. 문자 그대로의 순수 리듀서는 아님 |
| 1 / 4. 자연어→도구 / 도구는 구조화 출력 | **물려받음** | Claude Code / Codex 하네스가 제공, 재구현 안 함 |
| 11. 어디서든 호출 | **일부러 범위 밖** | 문 하나(런처)는 격리를 위한 선택이지 누락이 아님 |

아래에서 각 요소를 증거와 솔직한 빈틈까지 같이 짚어 봅니다.

## 드리프트리스가 잘 체화한 요소

### 2번 — 프롬프트를 직접 소유하기

> *12요소:* 프레임워크가 주는 기본값에 의존하지 말고, 프롬프트를 직접 관리하는
> 일급(first-class) 코드로 다뤄라.

이건 드리프트리스가 **무엇인가**의 거의 한가운데에 있어요. 에이전트의 행동은 어느
회사의 숨은 기본값이 아니라, 우리가 직접 소유하고 버전 관리하며 검토할 수 있는 작은
지시문 묶음이에요. 항상 로드되는 핵심 규칙은 `CLAUDE.md`(Claude)와
`AGENTS.md`(Codex)에 있고, 공통 절반 — 안전 계약, 증거 용어, 도구와 무관한 공통 스킬
— 은 `profiles/shared/`에 **한 번만** 있습니다. 두 도구가 같은 원본을 읽고, **거울
일치 게이트**가 두 프로필이 어긋나면 빌드를 실패시켜요.

드리프트리스는 "프롬프트를 소유한다"에서 한 걸음 더 나갑니다. *프롬프트를 어떻게
바꿀지에 대한 규칙*까지 소유해요. 지시문은 일부러 가볍게 유지하고(긴 절차는 항상
켜진 컨텍스트가 아니라 스킬과 스크립트로 보냄), 규칙 편집 점검표가 변경마다
hot(항상)/on-demand(필요할 때) 배치를 고르고 탈락시킨 대안을 기록하게 강제합니다.
프롬프트를 소유한다는 건 그 프롬프트의 식단까지 소유한다는 뜻이에요.

**판정: 잘 체화함.** 프롬프트는 코드이고, 단일 원본이며, 게이트가 두 복사본을
정직하게 유지합니다. 관련(영어판): [Single-source
mirror](../en/single-source-mirror.md).

### 3번 — 컨텍스트 창을 직접 소유하기

> *12요소:* 컨텍스트가 그냥 쌓이게 두지 말고, 모델에 무엇이 들어갈지 능동적으로
> 정하라.

드리프트리스는 컨텍스트를 채워야 할 양동이가 아니라 깎아 내려야 할 예산으로 봐요.
두 가지 장치가 이걸 구체적으로 만듭니다.

- **가벼운 핵심 컨텍스트.** 항상 필요한 규칙만 세션-핫 파일에 두고, 조건부인 건 전부
  필요할 때 부르는 스킬, `docs/`, 스크립트로 밀어냅니다. 다섯 축 경사 하강(자세히는
  [드리프트리스는 어떻게 학습하나](./드리프트리스는어떻게학습하나.md))은 **토큰**을
  *낮춰야 할* 축으로 직접 셉니다. 그래서 "한 작업에서 덜 읽기"는 분위기가 아니라
  측정되는 목표예요.
- **토큰 폭주 가드.** 긴 로그, 큰 JSON 덤프, 넓은 검색 결과처럼 큰 원본 출력은 증거
  파일로 빠지고, 살아 있는 컨텍스트로는 짧은 요약과 그 파일 경로만 돌아옵니다. 전체
  텍스트가 창을 가득 채우는 일이 없어요.

이건 12요소의 본능("모델이 보는 것을 직접 골라라")을, 손으로 프롬프트를 고치는 게
아니라 격리·비용 규칙으로 표현한 거예요.

**판정: 잘 체화함.** 컨텍스트는 정책과 폭주 가드로 큐레이션되고, 다섯 축 중 하나가
바로 "모델이 읽는 양"입니다.

### 7번 — 사람을 도구 호출로 부르기

> *12요소:* 사람에게 도움을 청하는 건 에이전트가 할 수 있는 정상적이고 구조화된
> 행동이어야 한다. 멈춰 버리는 것도, 우회하는 것도 아니다.

드리프트리스의 관리자 관계 전체가 이 아이디어 위에 세워져 있고, **작고 고정된 탈출
게이트 묶음**으로 날카롭게 다듬어져 있어요. 에이전트는 울타리 *안에서* 자율적이되,
특정 결정은 사람에게 되돌려야 합니다. 제품/우선순위 판단, 자격 증명, 결제/사용량
한도, 공개 출시, 되돌릴 수 없거나 파괴적인 일, 전역 승격, 사용자 데이터 이동,
force-push, 히스토리 초기화. 묻는 건 *오직* 이것들뿐이라 "사람을 부른다"가 불안한
습관이 아니라 정확한 도구예요.

그리고 그 연락 자체가 비개발자를 위해 설계됐어요. 아침 보고서는 쉬운 말로, 네 가지
라벨 중 하나로 시작합니다 — **완료 / 당신의 결정 필요 / 막힘 / 진행 중** — 그리고
명령어와 링크 같은 증거는 요약 *뒤에* 붙지, 요약을 대신하지 않아요. 탈출은 사람에게
돌려주는 깔끔하고 타입이 정해진 "반환값"이고, 그게 이 요소의 정신입니다.

**판정: 잘 체화함.** 사람 호출은 예외 경로가 아니라 정의된, 최소한의, 타입이 정해진
탈출이에요. 관련: [안전장치](./안전장치.md).

### 10번 — 작고 집중된 에이전트

> *12요소:* 모든 걸 하는 거대한 하나보다, 일이 분명한 좁은 에이전트를 선호하라.

드리프트리스는 거대한 하나가 아니라 조합형이에요. 두 층에서 드러납니다.

- **스킬이 작고 한 가지 목적이에요.** 티켓 닫기, PR 피드백 처리, 격리 검사 돌리기,
  관리자 브리핑 — 각각이 좁은 트리거를 가진 필요할 때 부르는 스킬이지, 모든 걸 하는
  거대한 프롬프트 하나가 아닙니다.
- **밤샘 작업은 부모 아래 좁은 워커들로 펼쳐져요.** 부모 세션이 할 일 목록을 살펴
  충돌이 적은 조각을 집중된 워커 세션들에 나눠 줍니다(동시 실행 수 제한, 각자 한
  가지 일). 에이전트 하나가 저장소 전체를 머릿속에 담으려 하지 않아요.

각 단위를 작게 유지하는 건 위의 "컨텍스트 소유"가 애초에 가능한 이유이기도 해요.
집중된 에이전트는 큐레이션할 창이 더 작으니까요.

**판정: 잘 체화함.** 스킬 층과 밤샘 오케스트레이션 둘 다, 거대한 하나보다 작고
집중된 여러 단위를 선호합니다.

## 드리프트리스가 부분적으로만 체화한 요소 (솔직한 빈틈)

### 9번 — 에러를 컨텍스트 창으로 압축하기

> *12요소:* 무언가 실패하면 *짧은* 요약을 모델에 다시 넣어 줘서, 전체 실패를 다시
> 읽느라 토큰을 태우지 않고도 회복하게 하라.

드리프트리스는 **압축 본능**은 있지만, 구조적 에러 회복 루프 전체는 아직이에요.

- *이미 하는 것:* 실패했거나 빈 도구 결과는 짧은, 타입이 정해진 상태로 다뤄집니다 —
  **UNVERIFIED, 절대 PASS 아님** — 그리고 빡빡한 규칙이 따라붙어요. 같은 호출을 1회만
  재시도하고, 그래도 비어 있으면 "전송 유실 UNVERIFIED"라고 말하고 **멈춥니다.** 원본
  실패를 쏟아붓거나 안 보이는 출력 위에서 맹목적으로 행동하지 않아요. 토큰을 낮추는
  다섯 축 압력과 폭주 가드도 에러 출력이 부풀지 않게 합니다. 그러니 에러는 *압축되고*,
  에이전트는 그걸 다시 읽느라 토큰을 *쓰지 않아요.*
- *빈틈:* 이건 "압축하고 안전하게 멈춘다" 규율이지, 12요소의 "에러를 압축하고
  루프 안에서 고쳐 나간다"가 아니에요. 드리프트리스는 단단한 실패 앞에서 일부러 *추론
  자동 재시도*보다 *멈추고 다시 확인*을 택해요. 비개발자의 저장소에서 되돌릴 수 없는
  행동 근처의 맹목적 재시도가 더 비싼 실수이기 때문이에요. 방어할 만한 절충이지만,
  그렇다고 구조적 에러요약-자가치유 루프가 아직 **없다**는 뜻이기도 합니다.

**판정: 부분적.** 에러는 압축되고 결코 조용히 삼켜지지 않지만, 자세는 안전-정지지
컨텍스트 안에서 반복하기는 아니에요. 앞으로의 개선: 작고 한도가 정해진 구조적 에러
요약을 만들어, 밤샘 워커가 *되돌릴 수 있는* 단계는 자동으로 다시 시도하되 되돌릴 수
없는 건 여전히 단단히 멈추게 하는 것.

### 5번 — 실행 상태와 업무 상태 통합하기

> *12요소:* 실제 데이터 모델과 어긋날 수 있는 별도의 숨은 "에이전트 상태"를 두지
> 말고, 둘을 통합하라.

드리프트리스의 "업무 상태"는 GitHub 이슈, PR, 그리고 날짜 찍힌 결정 등록부이고,
"실행 상태"는 세션이 지금 하고 있는 일이에요.

- *이미 하는 것:* 지속 상태는 휘발성 채팅이 아니라 **이슈/PR 본문**의 날짜 찍힌
  구역과 `docs/MANAGER_DECISIONS.md`에 삽니다. 매 세션이 일하기 전에 그 등록부를 다시
  읽으니, 에이전트의 작업 기억은 사람이 읽는 바로 그 기록에서 다시 끌어내져요.
  관리자가 못 보는 별도의 비밀 임시 상태가 없습니다 — 감사 흔적이 *곧* 상태예요.
- *빈틈:* 둘은 스키마를 가진 단일 기계검증 저장소가 아니라 **관습과 다시 읽기**로
  통합돼요. 밤샘 done-상태 게이트가 "완료" 주장 뒤에 미완 작업(lane)이 숨지 않았는지
  검사해서 이걸 *일부* 강제하긴 하지만, 실행 진행과 업무 기록은 여전히 규율 더하기 그
  게이트로 맞춰지는 두 표면이지 통합된 하나의 객체는 아니에요.

**판정: 부분적.** 상태는 지속적이고, 사람에게 보이고, 매 세션 다시 읽힙니다(강함).
하지만 문자 그대로의 요소처럼 단일 저장소가 아니라 게이트 + 관습으로 맞춰져요.

### 8번 — 제어 흐름을 직접 소유하기

> *12요소:* 모든 분기를 모델이 즉흥적으로 정하게 두지 말고, 에이전트 루프의 논리를
> 명시적으로 써라.

- *이미 하는 것:* 드리프트리스는 위험이 큰 분기를 모델이 마음대로 정하게 **두지
  않아요.** 탈출 게이트(7번), 교훈 승격 사다리, 그리고 격리/텍스트 안전/거울 일치
  게이트 스크립트는 모델이 말로 빠져나갈 수 없는, 명시적이고 결정적인 제어
  흐름이에요. 금지 경로는 *실패*, 비-ASCII 스크립트는 *실패*, 숨은 작업이 있는 "완료"는
  *실패*. 중요한 결정 지점에서는 제어 흐름을 직접 소유합니다.
- *빈틈:* *단계와 단계 사이* 오케스트레이션 — 한 작업 안에서 다음에 어느 도구를
  부를지 — 은 여전히 대체로 모델의 추론과 호스트 하네스(Claude Code / Codex)에 살지,
  드리프트리스가 소유한 루프 코드에 있지 않아요. 우리는 제어 흐름 위의 **가드레일**을
  소유하지, 제어 흐름 엔진 자체를 소유하진 않습니다.

**판정: 부분적.** 위험한 분기는 명시적으로 게이트되고 소유돼요. 잘게 쪼갠 루프는
하네스와 모델 몫이에요. 이건 의도된 분업(하네스를 다시 만들지 말 것)이지만, 완전한
소유는 아닙니다.

### 6번 — 간단한 API로 실행/일시정지/재개하기

> *12요소:* 에이전트는 진행을 잃지 않고 깔끔하게 멈췄다 다시 시작할 수 있어야 한다.

- *이미 하는 것:* 세션은 맨몸 재개 명령이 아니라 **런처 래퍼**로 재개돼요(아프게 배운
  교훈 — 맨몸 재개는 터미널 UI를 망가뜨릴 수 있었어요). 지속 진행 상태는 이슈/PR와
  결정 등록부에 살기에(5번 참고) 멈춰도 살아남고, 새 세션이 그걸 다시 읽어 실을
  이어 잡습니다.
- *빈틈:* 여기서 "재개"는 *지속 기록을 다시 읽고 이어 간다*지, 저장된 실행-상태
  토큰을 가진 일급 체크포인트/복원 API가 아니에요. 진행 중인 단계를 직렬화하는
  `pause()`나 그걸 바이트 단위로 복원하는 `resume()`은 없어요. 회복은 스냅샷-복원이
  아니라 재구성입니다.

**판정: 부분적.** 재개는 안전하고, 지속 상태 + 래퍼로 동작해요. 하지만 문자 그대로의
일시정지/재개 API가 아니라 재구성 기반이에요.

### 12번 — 에이전트를 상태 없는 리듀서로 만들기

> *12요소:* 에이전트를 입력 상태에서 출력 상태로 가는 순수 함수로 모델링해서, 행동이
> 재현 가능하게 하라.

이건 드리프트리스가 구성으로보다 *비유로* 가장 많이 맞는 요소예요.

- *이미 하는 것:* 밤샘 세션은 거의 상태 없는 것으로 다뤄져요 — 숨은 메모리 이월을
  믿는 대신, 매번 지속 입력(이슈, PR, 결정 등록부, 목표)에서 무엇을 할지 다시 끌어내고,
  결정은 다시 그 지속 기록으로 적어 냅니다. 쌓인 컨텍스트를 믿는 대신 매 세션 의도를
  다시 읽는 건 상태 없는 리듀서의 정신과 매우 닮았고, 그게 바로 "드리프트리스"가
  에이전트는 헤매지 않는다고 약속할 수 있는 *이유*예요.
- *빈틈:* 진짜 리듀서는 결정적이에요 — 같은 입력, 같은 출력. LLM 세션은 순수하지
  않고(샘플링, 모델 갱신, 도구 지연), 드리프트리스는 아닌 척하지 않아요. 우리는
  리듀서의 *규율*(믿는 숨은 상태 없음)은 얻지만 *보장*(결정성)은 얻지 못합니다.

**판정: 부분적, 비유로.** "지속 상태를 다시 읽고, 숨은 이월을 믿지 않는다"는 규율은
진짜이고 핵심을 떠받쳐요. 문자 그대로의 순수 함수 결정성은 주장하지 않습니다.

## 물려받았거나 일부러 범위 밖인 요소

### 1번 & 4번 — 자연어를 도구 호출로 / 도구는 그냥 구조화된 출력

이 둘은 모델 출력을 타입이 정해진 도구 호출로 바꾸는 *작동 방식*에 관한 거예요.
드리프트리스는 이걸 다시 만들지 않아요 — 호스트 하네스가 제공합니다(Claude Code와
Codex 둘 다 구조화된 도구 호출을 노출해요). 드리프트리스는 한 층 위에 앉아 *어떤*
도구가 허용되는지, *무엇*이 증거로 쳐지는지, *언제* 사람을 부르는지에 힘을 씁니다.
그래서 이 요소들은 **다시 만든 게 아니라 물려받은** 거예요 — 하네스를 다시 만드는 건
"엔진을 다시 짓지 말 것"을 어기는 일이니, 이게 맞습니다.

### 11번 — 어디서든 호출, 사용자가 있는 곳에서 만나기

12요소는 여러 진입점(슬랙, 이메일, 웹훅)을 권해요. 드리프트리스는 일부러 **문
하나**를 둡니다. 격리된 설정 홈을 고정하는 런처죠. 이건 빠뜨린 게 아니에요 — 여러
주변 진입점은 **격리**(감사 가능한 단일 진입점이 울타리 안에 두기 더 쉬움)와 비개발자
관리자 모델(자기 전에 붙여넣는 프롬프트 하나, 아침 보고서 하나)과 정면으로
충돌합니다. 그래서 이 요소는 **일부러 범위 밖**이고, 덮는 척하는 대신 그렇다고 이름을
댑니다.

## 솔직한 점수판

- **프롬프트를 소유하고, 컨텍스트를 소유하고, 작게 유지하고, 사람을 깔끔하게
  부른다** — 이건 드리프트리스의 핵심 강점이고, 그냥 의도가 아니라 게이트와 측정되는
  토큰 축으로 뒷받침돼요. (2, 3, 7, 10번.)
- **에러 압축, 상태 통합, 제어 흐름 소유, 일시정지/재개, 상태 없는 리듀서** —
  드리프트리스는 각각에 대해 *본능*과 부분적 장치를 갖췄고, 이름이 붙은 솔직한 빈틈이
  있어요. 빈틈의 반복되는 주제: 드리프트리스는 **가드레일과 지속 기록**은 소유하되
  **실행 엔진**은 호스트 하네스에 기대고, 되돌릴 수 없는 행동 근처에서 **자동 반복**
  보다 **안전 정지**를 택한다는 거예요. (5, 6, 8, 9, 12번.)
- **자연어→도구 / 구조화된 도구**는 하네스에서 물려받았고, **어디서든 호출**은 격리를
  위해 일부러 사양했어요. (1, 4, 11번.)

이 검토가 떠올린 가장 쓸모 있는 구체적 개선 하나: **9번** — 작고 한도가 정해진
*구조적 에러 요약*을 만들어, 밤샘 워커가 분명히 **되돌릴 수 있는** 단계를 짧은
실패로부터 자동 재시도하되, 되돌릴 수 없는 건 여전히 단단히 멈추게 하는 것. 그러면
12요소의 "에러를 압축하고 반복하라"와 드리프트리스의 "에러를 압축하고 멈춰라"가 가장
선명하게 갈리는 그 한 빈틈을, 안전 자세를 포기하지 않고 닫을 수 있어요.
