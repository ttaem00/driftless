# 코덱스와 클로드: 도구는 둘, 핵심은 하나

드리프트리스는 같은 저장소에서 AI 도구 **둘**을 돌립니다 — OpenAI Codex와
Claude Code. 둘은 서로 바꿔 쓰는 도구가 아니라, 각자 잘하는 대표 모드가 따로
있습니다. 드리프트리스는 일의 성격에 맞는 도구를 골라 씁니다. 그러면서도 그 밑에서
둘 다 **하나의 공유 핵심**을 읽기 때문에, 공유된 절반을 개선하면 두 도구가 동시에
좋아집니다.

이 문서는 그 갈림을 분명히 보여줍니다.

- [**Codex**가 드리프트리스를 쓰는 법](#codex가-드리프트리스를-쓰는-법) — 검토하고 출시하는 절반.
- [**Claude**가 드리프트리스를 쓰는 법](#claude가-드리프트리스를-쓰는-법) — 만들고 고치는 절반.
- [둘이 함께 읽는 **공유 핵심**](#둘이-함께-읽는-공유-핵심) — 한 번 고치면 둘 다 좋아짐.
- [**일에 따라 어느 도구를 쓸까**](#일에-따라-어느-도구를-쓸까) — 비개발자용 정리표.

> 정직한 전제: 지금은 **v0.1.0, 첫날**입니다. 아래 갈림은 두 프로필을 *어떻게 쓰도록
> 설계했는가*를 설명합니다. 자율 루프 자체는 원본 런타임이 자기 저장소에 돌렸고,
> macOS는 아직 UNVERIFIED(검증 안 됨)입니다
> ([호스트 검증 표](../en/host-evidence-matrix.md) 참고).

## Codex가 드리프트리스를 쓰는 법

드리프트리스에서 Codex의 대표 모드는 **goal 모드(목표 모드)** 입니다. 명확한 목표와
검증 가능한 성공 기준을 주면, Codex 세션이 그 목표를 향해 평소보다 더 오래 스스로
일합니다 — 단계마다 다시 물어보지 않고요. 이 모양은 저장소 유지보수의
**검토하고 출시하는** 쪽에 잘 맞습니다. 일의 범위가 분명하고 "끝났다" 신호가
구체적이기 때문입니다.

goal 모드가 잘 처리하는 일:

- **풀 리퀘스트(변경 제안) 검토** — PR을 읽고 성공 기준과 맞춰 보고, 무엇이
  통과인지 또는 무엇이 머지를 막는지 보고합니다. "끝났다" 신호가 구체적이라(머지
  가능, 검사 초록, 위험 게이트 없음) goal 모드에 딱 맞습니다.
- **이슈 분류(triage)** — 열린 이슈들을 훑어 혼자 해도 안전한 것을 가려내고,
  바로 할 수 있는 것을 처리합니다.
- **릴리스 게이팅** — 출시 전에 충족해야 할 조건을 검증합니다.
- **보안 및 유지보수 자동화** — 기준이 미리 적혀 있는, 반복 가능하고 잘 정의된 검사.

드리프트리스의 Codex 프로필이 이걸 가능하게 하는 방식:

- **goal-mode 스킬**(`profiles/codex/skills/goal-mode/SKILL.md`)이 계약을
  담습니다. 세 줄만 주면 됩니다 — 목표, 성공 기준, 범위/제외 — 나머지는 Codex
  세션이 소유합니다: 현황 조사, 레인 분해, 실행, 재시도, 검증/머지/이슈 게이트,
  마지막 관리자 보고까지. 진짜 관리자 결정만(제품 우선순위, 과금, 공개 릴리스,
  되돌릴 수 없는 행동, credential) 여러분에게 올라옵니다.
- **성공 기준 게이트.** goal 모드는 기준이 모호하면 길게 돌지 않습니다. 각 기준은
  *검증 가능*해야 합니다 — 통과하는 명령, 보이는 동작, 머지된 PR, 닫힌 이슈 —
  "잘 됐다" 같은 말은 기준이 아닙니다. 모호한 기준 + 긴 자율 = 시간과 토큰 낭비라,
  먼저 기준부터 구체화합니다.
- **병렬 안전 goal 레인.** 서로 독립적인 티켓 — 쓰기 영역이 겹치지 않고 선후
  의존이 없는 티켓 — 만 별도 worker goal 레인으로 펼칩니다(한 레인 = 한 이슈/브랜치).
- **세션 점유 핸드셰이크**가 두 레인이 같은 이슈/브랜치를 두고 경합하는 것을
  막습니다: *쓰기 전에 확인, 시작 시 점유, 충돌 시 직렬화, 끝나면 해제.* 오래된
  점유 표시 하나만으로는 "이 레인은 시도했다"가 성립하지 않으며, 쓰기 직전 재확인이
  작업 소실을 막는 핵심입니다.
- **빈 출력은 절대 PASS가 아닙니다.** 누락되거나 빈 결과는 성공이 아니라
  `UNVERIFIED`입니다 — 시스템 전체가 쓰는 동일한 증거 규율입니다.

> 비용 메모(정직하게): 더 길게 도는 Codex goal 레인 — PR 검토, 릴리스 자동화 —이
> API 크레딧을 쓰는 쪽입니다. 보고는 그 비용 축을 숨기지 않고 드러냅니다.

## Claude가 드리프트리스를 쓰는 법

드리프트리스에서 Claude의 대표 모드는 **ultracode(울트라코드)** 입니다 — 다이얼
두 개를 동시에 올린 것입니다: **최대(xhigh) 추론 노력** 과 **동적 멀티 에이전트
워크플로 오케스트레이션**. Claude가 레인을 계획하고 이 저장소 작업에 대해 자기
*자신의* 서브에이전트를 띄운 다음, 그 결과를 스스로 검토하고 채택합니다. 이 모양은
**만들고 고치는** 쪽에 맞습니다. 목표가 크거나 모호해서, 값싼 첫 시도가 틀렸을 때의
비용이 추가 사고 비용보다 큰 경우입니다.

ultracode가 잘하는 일:

- **크고 모호한 목표** — 하위 문제가 여럿이거나, 범위가 불분명하거나, 관리자가
  "다 해 / 전부 밀어 / 뭐가 중요한지 알아서 판단해"라고 한 경우. parent가 레인을
  계획하고, 실패한 레인을 복구하고, 보고 전에 정리합니다.
- **긴 컨텍스트 정리** — 여러 파일을 한꺼번에 가로질러 추론해야 하고, 전체 그림을
  눈에 담고 있는 것이 중요한 경우.
- **문서와 디자인 검토** — 설명을 쓰고 다듬기, 그리고 UI/레이아웃/사용자 흐름을
  바꾸기 전에 디자인 지침을 먼저 적용하기.
- **밤샘 루프** — parent가 독립 레인을 펼치고, 실패를 스스로 복구하며, 진짜
  관리자 결정만 위로 올립니다.

드리프트리스의 Claude 프로필이 이걸 가능하게 하는 방식:

- **ultracode 스킬**(`profiles/claude/skills/ultracode-orchestration/SKILL.md`)이
  *언제* 격상하고 *어떻게* 펼칠지에 대한 계약입니다. 규율은 **lean gradient(가벼운
  경사하강)**: 눈앞의 결정을 안정적으로 답하는 *가장 적은 노력*만 쓰고, 값싼 경로가
  틀릴 가능성이 클 때만 한 칸 올립니다. 작고 잘 정의된 일은 단일 에이전트 + 낮은
  노력으로 둡니다 — 작은 일을 과도하게 오케스트레이션하는 것 자체가 결함입니다.
- **하니스(harness)가 스폰을 소유합니다.** 세션 내 fan-out은 동시성 상한과 격리를
  내장한 네이티브 워크플로/서브에이전트 메커니즘을 거칩니다. Claude는 셸 호출을 한
  번에 묶어 병렬을 흉내 내지 않고(하나만 에러 나도 배치 전체가 취소됨), 외부
  런처로 빠져나가지도 않습니다.
- **서브에이전트는 이 저장소 작업을 하는 Claude 자신입니다** — 피어 AI도, 다른
  CLI도, 다른 모델로 가는 브리지도 아닙니다. 자기 오케스트레이션은 ultracode가
  바로 그것인 허용된 예외이고, 피어/재귀 외부 AI를 띄우는 것은 아닙니다.
- **날카로운 워크트리 격리:** *같은* 기존 파일을 고치는 레인은 격리 워크트리에서
  돌리고, *새 파일을 만드는* 레인은 옆 워크트리를 쓰면 안 됩니다(거기서 만든 새
  파일은 조용히 소실됨) — 대신 공유 체크아웃에서 겹치지 않는 파일 경로로 격리합니다.
- **parent가 모든 worker 출력을 다시 검증합니다.** "검토 준비됨(review-ready)"으로
  끝난 레인은 Done이 아닙니다 — parent가 diff와 증거를 검토하고, 게이트를 다시
  돌린 다음에야 채택/머지/닫음을 합니다.

## 둘이 함께 읽는 공유 핵심

Codex와 Claude는 **의도적으로** 서로 다른 패러다임이고, 드리프트리스는 한쪽이
다른 쪽을 흉내 내도록 억지로 맞추지 않습니다. Codex는 분리된 goal 레인으로 병렬화하고,
Claude는 하나의 하니스 안에서 네이티브로 병렬화합니다. 어느 쪽도 상대의 래퍼를
"빠뜨린" 게 아니라, 같은 일을 각자의 방식으로 풉니다.

두 도구가 두 벌로 두지 *않는* 것은, 진짜로 똑같은 부분입니다.

- **설계 계약** — 증거 어휘(PASS / FAIL / BLOCKED / UNVERIFIED / PARTIAL), 네 가지
  관리자 보고 라벨, manager-only(관리자 전용) 게이트;
- **forbidden-paths 스키마** — 두 containment 가드가 함께 읽는, 기계가 읽을 수
  있는 단일 안전 표면;
- **도구 무관 공유 스킬** — 충돌 매트릭스를 동반한 레인 분해, finish-to-done,
  밤샘 실행 번들.

그 공유된 절반은 `profiles/shared/` 아래에 **한 번** 살고, 각 프로필이 상대 경로로
그것을 읽습니다. 공유 규칙을 한 번 고치면 **두** 도구가 동시에 받아들입니다 —
어긋날 두 번째 사본이 없고, **mirror-parity 게이트**가 그 약속을 기계 검사로
바꿉니다. 도구별로 다른 절반(goal 모드 vs ultracode, `AGENTS.md` vs `CLAUDE.md`,
런처 메커니즘, 스킬 형식)은 다른 것이 *당연하고*, 억지로 맞추지 않습니다.

한 번 고치면 둘 다 좋아지는 미러가 정확히 어떻게 작동하고 게이트가 어떻게 강제하는지는
[단일 소스, 두 프로필](../en/single-source-mirror.md)을 보세요.

## 일에 따라 어느 도구를 쓸까

비개발자용 간단 정리표입니다. 보통은 손으로 고르지 않고 — 목표만 말하면 루프가
알아서 보냅니다 — 그 밑의 논리는 이렇습니다.

| 상황 | 손이 가는 도구 | 이유 |
|---|---|---|
| "머지하기 전에 이 PR을 검토해 줘." | **Codex** (goal 모드) | 범위 명확, 끝 신호 구체적. |
| "열린 이슈 분류하고 안전한 것만 처리해." | **Codex** (goal 모드) | 반복 가능, 기준이 미리 적힘. |
| "릴리스를 막을지 / 보안 검사를 돌려 줘." | **Codex** (goal 모드) | 통과 조건이 정의됨. |
| "백로그를 밤새 전부 처리해." | **Claude** (ultracode) | 크고 모호하고, fan-out parent. |
| "여러 파일에 걸친 이 지저분한 영역을 정리해." | **Claude** (ultracode) | 긴 컨텍스트 추론. |
| "문서를 쓰거나 고치고, 디자인을 검토해 줘." | **Claude** (ultracode) | 글쓰기와 디자인 판단. |
| "이 오타 하나만 고쳐 줘." | **둘 다, 낮은 노력** | fan-out도 xhigh도 과함. |

**한 줄 규칙:** 목표와 "끝났다" 신호가 이미 또렷하면 Codex, 목표가 크거나 흐릿해서
누군가 *무엇이* 끝인지부터 정해야 하면 Claude. 작고 뻔한 일이면 어느 도구도 격상하지
않습니다 — lean-gradient 규칙은 질문을 안정적으로 답하는 가장 적은 노력만 쓰라고
말합니다.
