# 쉽게 — 비개발자 관리자용 상황 브리핑

현재 작업/프로젝트 상태를 비개발자 관리자가 이해할 수준으로 설명한다. 관리자는 코드를 읽지 못하고, 영문 약어/기술 용어에 익숙하지 않다. 핵심은 "뭘 해달라는 건지", "뭘 확인하면 되는지".

> 관리자가 얻는 결과 (Manager outcome): 이 브리핑이 끝나면 관리자는 (1) 지금 무슨 일이 됐는지, (2) 무엇을 판단/승인해야 하는지, (3) 무엇을 어떻게 테스트하면 되는지를 코드 없이 바로 알 수 있어야 한다. 이 세 가지가 비면 브리핑이 덜 된 것이다.

증거 정직성은 그대로 지킨다: 실제로 실행/확인한 것만 ✅로 쓰고, 확인 못 한 것은 `확인 안 됨`/`자료 없음`으로 표시한다. 보고 첫 줄은 항상 쉬운 한국어 요약이고 명령/경로/원본 데이터/티켓 번호/에러 원문은 그 뒤 증거 줄에 둔다.

## 한눈에 (브리핑 순서)

아래 5개 섹션을 순서대로 채운다. 정보가 없는 섹션은 추측하지 말고 비우거나 `자료 없음`으로 표시한다.

1. 지금 뭘 하고 있었나 — 무슨 일을 하던 중인지.
2. 지금 어디까지 왔나 — ✅/🟡/⏳/❌ 체크리스트.
3. 관리자가 지금 판단해야 할 것 — 관리자 결정만, A vs B로.
4. 관리자가 테스트할 것 — 클릭/실행 단계 + 정상 기준.
5. 막혔거나 결정 보류 중인 것 — 있을 때만.

## 응답 구조 (이 순서 그대로)

다른 단계에서 "만든 것/확인한 것", "테스트/증거", "관리자가 직접 해볼 것", "막힘/미확인" 같은 구분을 쓰면 유지하되, 각 항목 첫 문장은 쉬운 한국어 요약으로 시작한다. 파일 경로, 명령, 원본 데이터, 티켓 번호, 에러 원문은 그 뒤 증거 줄에 둔다.

### 티켓/작업별 브리핑 변형

작업이 여러 개거나 "각 작업을 다시 출력" 요청이면 아래 틀을 작업 단위로 반복한다. 실제 정보가 없으면 추측하지 말고 `확인 안 됨` 또는 `자료 없음`으로 표시한다. 섹션 이름은 상황에 맞게 줄여도 되지만, 비개발자 관리자가 "왜 했고, 뭘 확인했고, 그래서 어떤 의미인지" 바로 보게 한다.

```markdown
## <작업 이름 또는 번호>

무슨 문제를 해결하기 위한 작업이었나:
- 사용자가 겪는 문제나 관리자가 기대한 결과를 쉬운 말로 1-2줄.

확인한 것:
- 실제로 확인한 증거만 적기. 실행, 화면, 테스트, 파일, 작업 상태 등.

결과나 한 일:
✅ 끝난 일:
- 눈에 보이는 결과 중심으로 작성.

🟡 남은 일:
- 아직 더 해야 하는 일. 없으면 생략.

한 일의 의미:
- 이 작업이 전체 목표에 왜 도움이 되는지 1-2줄.

결론:
- 완료 / 부분 완료 / 보류 / 실패 중 하나를 쉬운 말로.

막혔다면 어떤 게 막혔나:
- 막힘이 없으면 "없음". 있으면 원인, 영향, 다음 재시도 조건.

관리자 입장 의미:
- 지금 관리자가 판단·확인·승인해야 할 것. 없으면 "지금은 추가 판단 없음".
```

여러 작업을 한 번에 요약할 때는 각 작업 앞에 `✅ 완료`, `🟡 진행`, `❌ 막힘`, `⏳ 다음` 중 하나를 붙인다. 기술용어 제목보다 "관리자가 알아볼 이름"을 우선한다.

### 1. 지금 뭘 하고 있었나 (3-5줄)
- 이번 작업을 한 문단 요약.
- 코드/모듈 이름보다 사용자 눈에 보이는 동작.
  - 나쁨: "어느 파일의 어떤 함수를 수정"
  - 좋음: "영상이 끝날 때 자막이 잘리는 문제를 고치는 중이었어요"
- 중간 결과/실패 1줄.

### 2. 지금 어디까지 왔나 (체크리스트)
- ✅ 끝난 것 / 🟡 진행 중 / ⏳ 다음 차례 / ❌ 막힌 것 4 가지로 구분.
- 각 항목은 **관리자 시점**에서 의미 있는 단위. 코드 변경 단위가 아니라 "결과물" 단위.
- 관리자가 볼 수 있는 기능은 어디서 볼 수 있는지 함께 쓴다. 화면/메뉴/버튼/보고서 같은 진입점이 없으면 "아직 숨어 있음"으로 표시한다.
- 실험/시제품 결과가 좋아 보였으면 "도입함 / 다음 작업 / 폐기" 중 하나로 표시한다.
- 예:
  ```
  ✅ 어제 영상 자막 다시 생성 — 약어 잘못 읽던 문제 해결됨
  🟡 짧은 영상 후보 5개 자동 추리는 중 — 1개는 너무 짧아서 다시 뽑아야 함
  ⏳ 관리자 검토 페이지에 "거절 사유" 칸 추가 (다음 작업)
  ❌ 채팅 데이터 일부가 비어있음 — 외부 서비스 쪽 문제일 수 있음
  ```

### 3. 관리자가 지금 판단해야 할 것 (Decision Points)
- **개발 결정이 아니라 관리자 결정인 항목만** 추린다.
  - 관리자 결정: 채택/거절 기준, 톤·문체, 표기법, 우선순위, 어느 것부터, 공개 여부
  - 개발 결정: 어떤 라이브러리, 어떤 알고리즘, 코드 구조 — 이건 묻지 않는다
- 관리자에게 올릴 질문은 관리자만 결정할 수 있는 항목으로만 한정한다 (제품/우선순위, 로그인 자격, 결제/사용량, 공개 배포, 되돌릴 수 없는 작업, 사용자 데이터 이동 등).
- 각 결정은 **A vs B 형태로**, 각 선택의 결과를 1줄씩.
  - 좋음:
    > **Q. 자막을 큰 글씨 1줄 vs 작은 글씨 2줄 중 뭐로 할까요?**
    > - 1줄: 핸드폰 화면에서 잘 읽힘. 긴 말은 잘릴 수 있음.
    > - 2줄: 긴 말도 다 들어감. 1줄보다 글자가 작아짐.
- 결정이 없으면 이 섹션 생략. 억지로 만들지 말 것.

### 4. 관리자가 테스트할 것 (Test Steps)
- **클릭/실행 가능한 구체적 단계**. "확인하세요" 같은 추상 표현 금지.
- 각 단계마다 **무엇이 보이면 정상인지** 명시.
- 기능이 주소창에 직접 치는 주소, 숨은 경로, 내부 명령으로만 있으면 그대로 완료처럼 쓰지 말고 "아직 진입점 없음"과 후속 작업을 적는다.
- 예:
  ```
  1. 관리자 페이지 열기
  2. "후보 목록" 탭 클릭
  3. 어제 영상의 후보 5개가 보이는지 확인
     → 정상: 각 후보에 썸네일 + 30초 미리보기
     → 비정상: "후보 없음" 또는 페이지가 안 열림 → 알려주세요
  4. 첫 번째 후보 재생 → 자막이 영상과 맞는지 확인
     → 정상: 말하는 시점과 자막 시점이 ±1초 이내
  ```
- 테스트 환경 준비 명령이 필요하면 **복사 가능한 코드블록**으로 별도 표기.
- 이 작업 자체의 검사/머지 상태처럼 관리자가 직접 다룰 수 없는 항목은, 관리자가 직접 돌리게 시키지 말고 AI가 헬퍼 스크립트로 확인한 뒤 결과만 쉬운 말로 전한다. 어떤 검사가 통과했고 무엇이 남았는지 한국어 한 줄로.

### 5. 막혔거나 결정 보류 중인 것 (있을 때만)
- AI가 풀 기술 문제도 쓰되 "왜 관리자가 알아야 하는지" 1줄.
  - 예: "채팅 다운로드가 가끔 비어요. 원인 조사 중이라 그 영상 처리는 잠시 보류." → "그래서 그 영상 검토는 내일 이후"
- AI가 풀 수 있는 막힘이면 같은 작업 안에서 근본원인 후속 작업을 만들고 원래 검사를 다시 돌려 계속한다고 설명한다. 기술 로그 검토를 관리자에게 넘기지 않는다. 관리자에게 올리는 질문은 관리자만 결정할 수 있는 항목으로만.
- 구현/증명/검증 작업에서 "막힌 이유를 적어 뒀음", "한계를 기록했음", "후속 작업을 닫았음"을 완료처럼 말하지 않는다. 목표가 아직 안 되면 "아직 작동하지 않음 + 열린 후속 작업 + 다음 재시도 조건"으로 말한다.
- 관리자에게 보이지 않는 기능은 "숨은 기능/진입점 없음"으로 말하고, 만들어진 후속 작업을 알려준다.
- 긍정적인 실험 결과는 왜 의미 있는지 한 문장으로 풀고, 도입/후속/폐기 상태를 알려준다.

## 표현 원칙

- 금지 단어/표현: 사용자 눈에 안 보이는 내부 모듈 이름, 알고리즘 약어, 처리 단계 약칭 같은 기술 용어. 그리고 코드 작업 용어(커밋, 브랜치, 병합, 리팩터, 정규식, 파서, 스키마, 검증기, 설정 메타데이터 형식 등).
  - 꼭 필요하면 한국어 풀이 병기: "약어(쉬운 설명)".
- 비유 1줄 가능.
- 숫자는 의미 단위. "0.833 → 1.000" 같은 점수가 아니라 "10개 중 8개 잡던 게 10개 다 잡혀요"처럼.
- 이모지 OK(✅🟡⏳❌). 체크리스트/정상·비정상 마커에만.
- 40줄 이내 목표. 넘으면 §3·§4 중심으로 압축.

## 무엇을 절대 하지 말 것

- 코드블록만 던지고 "확인해주세요" 금지.
- 관리자에게 개발 결정 묻기 금지.
- 관리자에게 AI가 풀 수 있는 막힘 떠넘기기 금지. 원인 해결 후속 작업 만들고 계속 진행.
- "막힌 이유를 기록했으니 완료" 금지. 목표가 아직 작동하지 않는다고 말함.
- 관리자 눈에 보이지 않는 기능을 완료처럼 말하기 금지.
- 좋아 보인 실험을 다음 작업이나 폐기 이유 없이 "참고"로 묻기 금지.
- 진행 상황 과장/축소 금지. ❌ 숨기지 말고 ✅ 과장 금지.
- "성공적으로 완료되었습니다" 같은 빈 문장 금지. 무엇/어떻게/어디서 확인 가능한지 함께.

## 모르면 물어볼 것

상황 요약 정보 부족 시 추측 금지. 한 줄 질문 후 멈춤:

> "지금 어떤 작업을 하시던 중인지 한 줄로 알려주세요 — '어제 짧은 영상 검토하다 멈춤' 같은 식으로요."

## Sprint closeout summary

For bounded work, lead with four short lines: outcome, proof, manager-visible
check, and unresolved blocker. Put technical identifiers only in the evidence
section. Do not turn warnings or backlog ideas into current blockers unless they
directly invalidate the requested outcome.
