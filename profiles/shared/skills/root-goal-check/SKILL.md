## Improvement Principle

Use root-cause analysis and root-cause fixes, not symptom patches. Generalize as principle-based guidance or design principles; avoid spec/case overfitting and special-casing unless evidence proves a bounded exception reduces user effort, maintainer effort, maintenance risk, or safety burden.
# Root Goal Check (shared)

두 프로필이 함께 소비하는 tool-agnostic 게이트다. 새 아이디어·외부 자료·기술 제안·
백로그를 본질 목표와 비교해 **도입 / 보류 / 폐기**를 판단한다. 도입 전에 통과해야 한다.

## 본질 목표
비개발자 관리자가 손대지 않아도 두 격리 프로필이 shared tier를 자동·안전·일관되게
소비하는 제품. 최상위 목표는 **시간 / 토큰 / 사람 개입을 줄이는 경사하강 최적화**.
정액제 구독을 전제로 하므로 비용은 달러가 아니라 사용량(요청/세션/토큰)으로 따진다.

## 판정 질문
1. 이 아이디어/자료가 위 본질 목표를 **직접** 전진시키나? 아니면 안전·유지보수·
   증거·백로그 전제일 뿐인가?
2. 도입 비용(시간/토큰/핫컨텍스트/관리자 부담)이 막아 줄 실수·재작업보다 작나?
3. containment 또는 정본(single-source) 모델을 위반하나? (외부 레포 쓰기, peer/
   다중 에이전트 오케스트레이션, 호스트 전역 설정 침범, peer/재귀 AI 호출은
   기본 폐기 대상이다.)
4. hot(기본 로드)인가 on-demand인가? hot이면 작고 정당해야 한다.

## 폐기(KILL/reject) 전 필수 점검 (비싼-프레임 오판 방지)
비용을 이유로 무언가를 폐기하기 전에 반드시:
- **lean=pilot, not discard:** "무거워 보인다"는 폐기 사유가 아니다. lean은 작게
  pilot으로 도입해 실사용으로 측정하고 가치 있으면 우리식 최소 변형으로 들이는 것.
  무거운 후보일수록 먼저 **가장 작은 pilot 형태**를 설계해 돌려보고(예: 지식그래프
  -> 레포 구조 1장 요약 파일, 통합상태 DB -> 기존 issue/PR/evidence를 읽는 작은 인덱스),
  5축 이득이 측정으로 없을 때만 폐기. pilot 없이 인상만으로 폐기하면 회피다.
- **가장 싼 형태 질문:** "같은 목표를 LLM 호출 없이 / 더 싸게 달성하는 형태가 있나?"
  (예: 스킬 최적화의 싼 형태 = LLM 호출 0의 정적 검증 게이트; 자동 요약 = 세션당 1회.)
  비싼 해석 **하나만** 측정해 KILL하지 말 것.
- **출처-프레임 명시:** 지금 이 아이디어를 어느 렌즈로 보고 있나(ML 제품? 검증 도구?).
  같은 이름이라도 출처가 프레임을 가른다(예: 공유메모리 제품 = 비싼 자동화 vs 검증
  하니스 = 싼 게이트). 다른 렌즈를 1개 이상 확인하고 적을 것.
- **선례 조사:** 같은 목표를 이미 더 싸게 푼 선례(이 레포의 닫힌 이슈/PR 등)가 있나?
  있으면 그 프레임을 기본값으로 삼는다.

## 결정 (하나)
- **도입(adopt):** 본질 목표 전진 + 비용 정당 + 모델 위반 없음 -> 어느 tier(shared/
  프로필별)에 둘지 명시 + 실행 티켓.
- **보류(defer):** 가치는 있으나 지금은 아님 -> 측정 가능한 재평가 트리거와 함께 백로그 등록.
- **폐기(reject):** 목표 불일치 또는 모델 위반 -> 근거와 증거를 남긴다(되살아나지 않게).

## 외부 자료 특칙
외부 레포는 관리자가 승인한 범위에서만 read-only로 참고한다. peer/다중 에이전트
오케스트레이션은 정본에서 폐기 대상이므로 자동 reject. 도입 후보는 "여기서 직접
빌드·실행하기 전에는 proven 아님"으로 다룬다(동작 주장은 UNVERIFIED).
